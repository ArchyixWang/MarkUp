# MarkUp（马克派）前端设计风格规范

**创建日期**：2026-05-23  
**适用范围**：`apps/web` 前端应用，包括认证页、登录后主工作台、任务/标注/审核/企业模块、用户账号管理页面。
**维护目的**：将当前前端视觉基线、登录后工作台风格和账号管理页面布局约束文档化，避免后续页面各自发挥导致产品风格割裂。

## 1. 总体定位

MarkUp 是面向企业协作的数据标注平台，前端应优先体现以下气质：

- 简洁、稳定、可信赖：页面为重复性生产工作服务，避免营销页式的大面积装饰。
- 高信息密度但不拥挤：任务、题目、审核、质量、AI 预审结果都应可快速扫描。
- 清晰的状态与操作层级：待处理、进行中、异常、通过、打回等状态必须一眼可辨。
- 面向数据生产链路：主界面围绕任务、数据集、模板、标注队列、审核队列、质量指标、企业协作和 AI 资源展开。

参考成熟数据标注平台时，吸收其工作台思路而非照搬视觉：导航稳定、任务列表清晰、关键指标前置、表格与筛选能力完整、审核和质量状态突出。Xpert Studio 等专家数据/标注平台的公开定位强调面向大模型和 Agent 的专家级数据生产，MarkUp 的 UI 也应服务于“高质量数据生产链路”这一核心目标。

## 2. 当前前端风格基线

当前前端已引入 Ant Design 作为组件库基线，`apps/web/src/main.tsx` 通过 `ConfigProvider` 注入主题与中文 locale。`LoginPage` 保留品牌入口风格：

- 结构：左侧品牌介绍与能力指标，右侧认证卡片。
- 气质：温暖、轻量、带有产品首次进入时的品牌感。
- 颜色：暖白、陶土橙、苔绿色、深墨色，来自 `LoginPage.css` 中的 `--paper`、`--clay`、`--terracotta`、`--moss`、`--ink` 等变量。
- 组件：大标题、轻量玻璃卡片、分段登录/注册 tab、圆角输入框、主操作按钮、提示条、第三方登录按钮。

这套风格适合作为未登录品牌入口，不应直接复制到登录后主界面。登录后页面需要更克制、更工具化，减少大标题、玻璃拟态和大面积渐变背景。

## 3. 登录后主界面风格

成功登录后的主界面采用简洁的橙白蓝色调：

| 角色 | 建议颜色 | 用法 |
| --- | --- | --- |
| 主背景 | `#F7F9FC` / `#FFFFFF` | 页面背景、内容区、表格底色，保持干净。 |
| 主文本 | `#172033` | 标题、表格主字段、关键数字。 |
| 次级文本 | `#667085` | 描述、说明、元信息、空状态补充文案。 |
| 主品牌橙 | `#F97316` | 主要 CTA、当前高亮、关键提示、MarkUp 品牌识别。 |
| 浅橙底 | `#FFF4E8` | 待处理提醒、轻量标签、激励/奖励信息。 |
| 信息蓝 | `#2563EB` | 链接、数据洞察、AI 辅助、进度与可点击分析入口。 |
| 浅蓝底 | `#EFF6FF` | 信息提示、AI 预审摘要、看板小组件背景。 |
| 成功绿 | `#16A34A` | 通过、完成、健康状态。 |
| 警告黄 | `#D97706` | 待补充、待复核、预算接近上限。 |
| 危险红 | `#DC2626` | 错误、打回、删除、失败状态。 |
| 边框 | `#E5E7EB` | 卡片、表格、分割线、输入框默认边框。 |

橙色承担品牌和主行动，蓝色承担信息、链接和 AI/数据洞察。不要让页面被单一橙色或单一蓝色铺满；白色和浅灰背景应占主要面积。

## 4. 主工作台布局

登录后页面采用 SaaS / 数据平台式布局：

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部栏：Logo / 企业切换 / 全局搜索 / 通知 / 用户入口        │
├──────────────┬─────────────────────────────────────────────┤
│ 侧边导航      │ 页面标题 + 操作区                            │
│ Dashboard    │ 筛选 / 指标 / 表格 / 工作台主体                │
│ Tasks        │                                             │
│ Datasets     │                                             │
│ Templates    │                                             │
│ Labeling     │                                             │
│ Review       │                                             │
│ AI Resources │                                             │
│ Team         │                                             │
│ Settings     │                                             │
└──────────────┴─────────────────────────────────────────────┘
```

布局约束：

- 顶部栏固定高度，建议 56-64px；不要放大成营销页导航。
- 顶栏登录态在 `工作台` 按钮左侧提供唯一的个人信箱入口。入口使用图标按钮和未读角标，点击后只打开轻量概览窗口；概览只允许刷新、标为已读、星标/取消星标和进入完整页，不提供删除或批量管理。已过期通知仍可在概览中作为历史消息展示，但不计入未读、不显示单条已读动作或未读徽标。完整信箱通过概览中的“查看全部”进入工作台隐藏页面，不加入侧边栏、账号页或企业管理导航。
- 侧边栏宽度建议 220-248px；导航项使用图标 + 文本 + 可选数量徽标。
- 工作台侧边栏必须按当前用户身份和权限动态生成。企业用户显示与 Team Admin / Owner / Reviewer / Agent 对应的数据生产、审核质检、企业管理和个人工具入口；Labeler 默认进入 `labeler-dashboard`。个人 Labeler 只显示个人标注工作台与个人工具；企业内 Labeler 显示企业项目、账号管理、企业信息和个人工具，不展示人员管理、资源配置、公告治理或操作日志入口，侧栏文案使用“企业项目 / 我的项目 / 项目历史”，不得使用自由接单语义。
- 公告通知页必须按权限降级：Reviewer 等只具备 `team:read` 的成员可读取通知列表、标为已读和维护个人通知设置，但不得触发成员列表、接收人预览、新建企业通知、撤回或删除企业通知等需要成员管理权限的操作。
- 主内容区使用剩余浏览器视口作为工作区，页面宽度较大时优先增加内容密度，不做居中窄版落地页。
- 页面标题区包含标题、简短说明、主操作按钮；说明文字不超过一行半。
- 工作台面包屑和每个功能页的页头应固定在内容滚动区顶部。面包屑使用实色背景、浅灰下分割线和极轻阴影；页面标题下方内容滚动时，标题不能随列表、表格或表单一起离开视口。
- 个人信箱完整页属于工作台内隐藏页面，面包屑显示 `工作台 / 个人信箱`，但侧边栏不展示入口；避免个人消息能力和企业公告通知治理页混淆。完整页使用 Ant Design `Tabs + Table rowSelection + Drawer`，类型 Tab 从后端 `type_options` 动态生成，并额外提供 `未读`、`星标`、`已过期`状态视图；支持搜索、单条/批量已读未读、星标/取消星标和个人软删除。已过期通知仍可查看详情，但不计入未读、不展示处理操作。企业广播统一显示为 `企业通知`，Labeler 个人信箱不展示企业广播 Tab。
- 系统主动发出的任务、审核、导出和权限安全重要节点仍复用同一个个人信箱入口；详情 Drawer 可展示 `metadata` 中的业务状态或失败原因摘要，并通过 `action_url` 提供“前往处理”跳转。导入完成、模板发布成功、领取成功、提交成功、钱包成功变更等确认型流水不进入信箱，前端不得为系统通知新增侧栏、账号页或组织页入口。
- 页面切换时面包屑应保持稳定，不出现先清空再重绘的闪烁；详情页、设计器等动态尾部面包屑必须只在所属父页面显示。
- 复杂页面使用「固定页面标题 + 固定指标概览 + 固定搜索筛选 + 内部滚动表格/工作区 + 详情抽屉」组合。
- 详情、编辑、审核意见优先使用右侧抽屉或独立页面；避免在列表中塞入长表单。

固定视口工作台约束：

- 工作台页面在桌面端必须固定在一个浏览器视口内，禁止依赖浏览器页面上下滚动承载主流程。`AppShell`、面包屑、页面标题、数据概览、搜索筛选栏都保持稳定位置。
- 页面主内容只能在明确的业务容器内部滚动，例如 Ant Design `Table` 的 body、Designer 画布、属性面板、右侧 `Drawer`、Tabs 内容区或表单内容区。
- 页面不得出现整体横向滚动。表格列过多时，优先减少主表列、使用 `ellipsis + Tooltip`、把低频字段收进详情 `Drawer` / `Descriptions`，不要让用户左右拖动页面来完成核心操作。
- 详情 Drawer 必须遵循“外壳固定、内部滚动”规则：标题、步骤条和底部操作区保持稳定，正文由明确的内容容器承担纵向滚动；不能因为额外的 `overflow: hidden` 或高度链路缺失而截断内容。
- 任务管理、模板搭建、数据集管理、人员管理、公告通知、操作日志等列表页统一采用：页面标题栏 -> 数据概览栏 -> 搜索筛选栏 -> Ant Design `Table`。这四段从上到下紧凑排列，前三段固定，表格数据区内部滚动。
- Owner 生产类列表页，尤其任务管理、模板搭建、数据集管理，必须复用同一套列表骨架：`production-list-page` + Ant Design `Statistic` 统计条 + 单行 `Input.Search` / `Select` / `Button` 筛选操作条 + `Table` 表格容器。统计条必须紧贴页面标题栏，筛选条必须紧贴统计条，不允许在中间插入自定义 section 标题或页面说明。
- 任务管理、模板搭建、数据集管理允许在筛选栏右侧提供 Ant Design `Segmented` 视图切换：`表格` 为默认视图，`卡片` 为扫描视图。切换控件必须和搜索、筛选保持同一行，不改变固定标题、统计、筛选和主体容器高度。
- 人员管理、公告通知、操作日志等治理列表页同样复用这套骨架；如操作日志筛选项较多，可保持同一筛选栏视觉但增加栏高或折叠高级筛选，不得把筛选条塞进表格容器内部。
- 搜索和筛选必须保持单行稳定布局。筛选下拉不得改变页面 `scrollWidth` / `scrollHeight`，不得触发浏览器窗口抖动；下拉容器按组件局部容器或稳定门户挂载，不能导致页面级滚动条出现或消失。
- 任务管理及新建/修改任务子页面使用浅蓝色作为模块主色，橙色不再作为该页面主强调色。任务管理的新建/修改子页面必须使用独立双栏工作区：顶部固定标题栏，下方固定水平步骤条，左侧显示当前步骤的完整配置，右侧显示发布摘要；步骤区和摘要区各自内部滚动，不依赖浏览器页面滚动承载流程。
- 新建任务子页面标题栏不放常驻“保存草稿”按钮；填写有效内容后通过防抖自动保存，底部可保留低权重手动保存入口。基础信息不预填演示文案，必须为空白待用户输入。
- 新建任务步骤条中不再使用“审核与资质”合并步骤：`选择模板` 与 `绑定数据` 合并为 `模板与数据`，`分发策略` 与 `积分奖励` 合并为 `分发与奖励`；资质要求放入 `分发与奖励`，审核拆成 `AI 预审` 和 `人工复审`，并新增 `用户协议` 步骤。用户协议步骤使用 Ant Design `Switch`、`Checkbox`、`Input.TextArea` 和 `Upload`，支持默认协议模板、文本编辑和文件选择；任务广场详情抽屉领取流程必须使用 `任务详情 -> 签署协议 -> 领取确认` 三步，第二步显示任务用户协议，未勾选必签协议不允许进入领取确认。
- 新建任务右侧发布摘要必须避免裸数字：积分为 0 或未填时显示 `未填写`；通过率、AI 阈值等百分比要写明业务含义并带 `%`，例如 `最低准确率 80%`、`通过阈值 90%`。
- 企业信息和资源配置不强行表格化，但仍遵守固定视口：摘要区固定并优先复用 Ant Design `Statistic` 统计条，表单、Tabs 或配置表在内部滚动区承载。
- 页面级操作反馈使用 Ant Design 原生反馈组件，不得自造浮层。生产类页面中的短时成功/失败反馈统一使用顶部 `message`，默认 5 秒自动消失，不在标题、统计和筛选之间插入会撑开布局的内联提示。
- 数据集管理、模板搭建、任务管理这三类页面的操作结果提示统一规则如下：成功/失败轻提示走顶部 `message`；必须用户确认的危险动作继续使用 `Modal.confirm`、`Popconfirm`；长表单和详情编辑继续使用 `Modal`、`Drawer`，不强制 5 秒自动关闭。
- 页面空态、权限不足和阻断性说明不走顶部 `message`，统一使用页面内 `Alert` 或 `Empty`；模态内部的格式说明、导入说明、字段说明统一使用模态内 `Alert`，不与页面级反馈混用。
- 数据集管理、模板搭建、任务管理中的 `created_at`、`updated_at` 等生产时间统一通过前端时间格式化函数展示：后端若返回带 `Z` 或时区偏移的 ISO 字符串，按原时区解析；若返回无时区的 ISO datetime 字符串，按 UTC 解析后再转换为浏览器系统时区显示，避免最近更新时间与系统时间相差 8 小时。
- 视觉区块以浅边框和白底为主，避免页面 section 全部做成浮卡；指标概览更像状态条，不做大面积卡片墙。

## 5. 核心页面风格

### 5.0 公开首页

- 公开首页 `/` 是品牌入口和双角色分流页，当前采用 AI-first 定位：H1 保持短标题，AI 辅助发布、模板搭建、AI 预审、人工复核和导出闭环放在副文案、产品预览和下方能力区展开。
- 首屏主视觉应呈现 MarkUp 风格的产品场景预览，优先展示 AI 预审评分、风险标签、审核维度矩阵、人工复核状态和导出就绪进度；不直接使用带旧品牌痕迹的需求示意截图。
- 首屏 CTA 保持企业 Owner 与标注员双角色平衡：企业侧主按钮打开注册弹层，标注员侧进入公开任务广场 `/tasks`。
- 首页可以比登录后工作台更有品牌感，但仍要服务数据生产链路，避免纯营销文案、抽象装饰和大面积不可解释视觉。
- 首页实现优先使用 Ant Design `Button`、`Card`、`Tag`、`Statistic`、`Progress`、`Alert` 和 `Steps`，图标使用 `@ant-design/icons`；移动端按钮单列、AI 预览降密度，不出现横向滚动或文字重叠。
- 公开外部页面 `/`、`/tasks`、`/solutions`、`/help` 统一使用共享 `PublicFooter` 页脚。页脚基于 Ant Design `Layout.Footer`，内容版心不超过 1320px，包含品牌说明、公开导航和平台生产链路标签；不得出现在 `/workspace`、`/platform`、`/onboarding`、OAuth callback 或指派任务落地页中。

### 5.0.1 公开套餐方案页

- 公开导航中的原 `发布任务` 销售入口调整为 `解决方案`，目标路由为 `/solutions`；历史 `/publish` 仅作为兼容入口跳转到 `/solutions`。工作台内 `workspace?page=publish-task` 仍是企业端真实任务发布向导，不改变发布状态机、权限或 API。
- `/solutions` 是公开套餐方案页，面向 Owner、Team Admin 和采购决策人；首页已经承担产品介绍，本页不再重复完整销售叙事。
- 页面主线为 `轻标题区 -> 套餐卡 -> 最终 CTA`：标题使用 `企业套餐方案`，主 CTA 使用 `创建企业账号`。
- 页面必须展示企业会员套餐 `Free / Basic / Pro / Enterprise / More`，套餐口径与 `docs/api/team-profile.md` 保持一致；卡片展示适合阶段、年费、成员上限、活跃生产任务、数据集存储和 CTA。
- 顶部轻标题区可以使用浅色品牌面板和套餐总览面板增强公开页质感；套餐卡需通过色带、价格区、推荐标识和清晰权益层级强化可扫读性。
- Pro 高亮推荐；More 仅作为定制入口展示，不伪造自动购买流程。用户界面不得出现内部定位说明、设计备注或对页面用途的自我解释。
- 不再单独展示流程画布、生产工作流、FAQ、套餐对比表或额外选择指引；套餐差异由卡片承担。
- 实现优先使用 Ant Design `Button`、`Card`、`Tag` 等基础组件，图标使用 `@ant-design/icons`；移动端单列布局，套餐卡片不得造成页面级横向滚动。

### 5.1 Dashboard

- 企业用户 `/workspace` 默认主页面已接入企业 Dashboard 聚合接口：先读取 `/teams/admin/overview` 默认企业，再读取 `/teams/{team_id}/dashboard`。Labeler 默认进入独立 `labeler-dashboard`，按当前身份分为企业内 Labeler 的 `企业项目工作台` 和个人 Labeler 的 `个人标注工作台`。
- 顶部固定标题栏使用中文 `企业工作台`，展示企业名称、当前企业角色 Tag、生成时间、刷新按钮和当前角色主行动按钮；保持工作台固定标题栏风格，不做营销型 Hero。
- Dashboard 不再使用横贯全宽的待办/风险条、Alert 条或 Chip 条作为首屏模块；待审核、AI 异常、导出失败、余额和额度风险必须并入 KPI 状态条、图表摘要或角色主行动，避免在标题和指标之间插入突兀分隔。
- 指标状态条使用 Ant Design `Statistic` 承载生产、审核、AI、导出、钱包和成员额度指标；卡片半径不超过 8px，浅边框、弱阴影或无阴影。
- 主体升级为图表化企业驾驶舱，不改变 `GET /api/v1/teams/{team_id}/dashboard` 响应结构，所有图表均从现有聚合字段派生。生产漏斗/生产转化使用稳定的横向阶段条展示题目总量、已领取、已提交、已通过，避免默认 Funnel 或横向 Bar 轴标签在小容器中失真；任务状态使用 Ant Design Charts `Column` 且过滤 0 值状态，避免出现不适合计数的小数轴标签和空柱；审核结果使用 `Pie`；AI/导出状态使用 `Column`；会员资源额度使用紧凑 quota progress，存储单位格式化为 GB/TB 并保持首屏可见。
- 最近活跃任务、最近导出、最近 AI 预审、通知和审计继续使用 Ant Design `Table` / 项目 `EnhancedTable`，不再用松散 `List` 承载核心记录。桌面端标题栏固定、Dashboard 内容区内部滚动；图表容器必须有稳定高度，窄屏下单列堆叠且不得出现页面级横向滚动。
- Dashboard 首屏必须采用密铺网格：KPI 状态条、生产转化图、任务状态图和资源额度之间使用统一紧凑间距；图表带内的卡片必须同高填满网格单元，不允许右侧短卡片在首屏留下大块空洞或因数据加载造成位置跳动。
- 角色降级遵循后端 `shortcuts` 和可见数据：Reviewer 聚焦人工审核与通知，不展示任务生产管理主入口；Agent 只作为企业 Dashboard 内的 AI/资源降级视角，不新增独立 Agent 看板页面；Owner 聚焦生产与导出；Team Admin 显示全量治理入口。

### 5.1.1 Labeler Dashboard

- 企业内 Labeler 看板标题固定为 `企业项目工作台`，数据源为 `GET /teams/{team_id}/labeler-dashboard`。首屏聚焦公司项目：公司分配任务、待标注题目、待修改题目、待审核提交、已通过、项目完成率和返工率；图表展示公司项目进度分布、提交状态分布和质量分布；列表展示我的公司项目、最近提交记录和企业通知。
- 企业内 Labeler 看板不得在首屏展示公开任务广场推荐、高收益任务、自由接单赚钱等个人平台语义；积分收益如后续出现只能作为低优先级个人工具，不作为核心 KPI。
- 个人 Labeler 看板标题固定为 `个人标注工作台`，数据源为 `GET /profile/dashboard`。首屏聚焦个人任务和成长：已领取任务、待标注、待审核、已通过、可用积分、本月收益、信誉分、资质状态；保留任务广场推荐、积分管理和资质认证入口。
- 两类 Labeler 看板都必须使用固定页头、密铺 KPI、紧凑图表网格和内部滚动表格。表格操作列继续复用 `WorkspaceTableActions`，主要文本优先显示任务名、题目序号、提交状态、更新时间和短编号，不把长 `task_id/submission_id/question_id` 作为主展示。

### 5.2 任务 / 数据集 / 模板列表

- 以表格为主，支持关键词搜索、状态筛选、角色筛选、时间排序。
- 行内展示状态标签、进度条、题目数、提交数、通过率、最近更新时间。
- 主操作使用橙色按钮，如“创建任务”“导入数据集”；次级操作使用白底蓝字或灰色按钮。
- 批量操作出现后放在表格顶部工具条，避免每行重复过多按钮。
- 企业端生产页的主工作区应在剩余视口内企业高频操作，长列表、属性面板和配置区必须使用内部滚动；历史列表、已保存模板、低频长内容也应进入固定工作区内的 Tabs、Drawer 或内部滚动容器，不再让浏览器页面自然纵向滚动。
- 数据集导入、新增渲染变量等短表单优先使用按钮触发弹窗，不在主页面长期展开，主页面保留表格、字段账本、预览和摘要等高频扫描区域。
- 模板搭建页的 Renderer 预览使用独立弹窗或抽屉承载，避免挤在右侧属性面板；右侧面板只保留当前组件的基础属性、校验、联动和 AI 配置。
- 模板 Designer 子页面必须固定在一个浏览器视口内：标题栏和结构状态条固定，左侧物料栏、中间画布、右侧属性面板分别使用内部滚动；模板名称与参考数据集保持同一行紧凑展示。
- 模板 Designer 顶部的“参考数据集”应和“模板名称”一样使用固定标签 + 输入控件的紧凑结构，不把标签文案写进 Select placeholder，避免聚焦或展开时标签上跳。
- Designer 物料、页签和画布操作优先使用 Ant Design 图标按钮、Button、Tooltip、Dropdown 和 Select/Input。物料区用图标 + 简短名称扫描，不堆叠长说明；页签重命名、复制、删除、左右移动在页签标题更多菜单完成，不再保留独立“页签操作”按钮；组件上移、下移、复制、删除使用图标按钮。
- Designer 顶部标题栏操作常态只保留图标按钮并通过 Tooltip 命名，包括返回、Renderer 预览、导出 schema、保存草稿和发布模板。
- 模板 Designer 也需要和任务发布页一致的自动保存逻辑：有有效修改后防抖自动保存为 `draft + auto_saved=true`；手动保存后如果未继续修改，不应再次触发自动保存。模板列表与 Designer 结构状态条需要区分“草稿”和“自动保存”。
- Designer 中间画布应呈现为一张纯白考试/问卷页面，组件以紧贴的题目切片/分段方式排列，不做圆角矩形卡片堆叠；组件之间没有可见空隙，插入入口以绝对定位覆盖在题目分割线上，不占用题块上下排版高度，通过 Ant Design `Dropdown` 选择物料，下拉菜单以分割线中心对齐且不得被画布滚动容器裁切。Designer 可在画布顶部提供小型折叠按钮，收起标题栏与结构状态条以扩大画布高度；左右栏标题必须 sticky 固定在栏顶部。
- 模板画布的插入入口应与组件选中态区分：组件悬停只显示当前组件操作，不触发相邻插入线；插入槽只在鼠标接近分割线、获得焦点或拖拽指向时显性显示；从物料栏或画布拖拽组件时，只有拖到蓝色分割线热区才显示投放高亮，拖到组件块本身不得高亮组件或触发中间插入，分隔线生效时下方组件块也不得显示高亮背景。最后一个组件后方必须保留独立蓝色分隔线热区，用于追加到画布末尾；拖拽悬停阶段不做物料上下偏移动画，释放后仅让实际落位物料用 `transform + opacity` 滑入最终位置。组件选中不得常态显示蓝色高亮，点击画布空白区域应取消画布选中态。
- Renderer 预览页复用 Designer 的固定视口、图标化操作和“收起标题与概览”模式；收起后只保留预览数据选择工具条和 Renderer 主工作区。
- 减少若干个圆角矩形并排的磁贴式排版；指标和摘要优先做成状态栏、表格行、侧栏定义列表或分隔线信息组。

### 5.3 标注工作台

- 左侧或顶部展示任务上下文、题目列表和进度。
- 主作答区保持白底，输入控件边界清晰，减少装饰背景。
- 右侧可放 AI 辅助、质检提示、打回说明、历史轮次。
- 自动保存状态必须可见但不干扰，建议放在标题区右侧或底部状态栏。

### 5.4 审核工作台

- 使用双栏或三栏结构：队列、提交内容、审核意见/AI 预审结果。
- AI 预审摘要用浅蓝底，人工打回/风险提示用浅橙或浅红底。
- 通过、打回、要求修改等动作必须固定在易触达区域，危险操作与普通操作视觉区分。
- 多轮 diff 应使用左右对比或高亮变更，不要仅用文字描述。

### 5.5 企业与 AI 资源

- 企业成员页面参考企业后台管理：表格、角色标签、权限摘要、邀请状态、批量管理。
- AI 预审页属于审核质检控制台，不做概览卡片墙或松散说明页。当前实现已从旧 job 队列页升级为“按任务组织的 AI 预审工作台”：首页一级对象是任务，默认使用 Ant Design `Table` 展示任务名称、任务状态、AI 配置、覆盖率、状态分布、建议分布和最近更新时间，并提供 `Segmented` 切换卡片扫描视图；旧 job 队列接口仅保留兼容，不再作为首页心智。
- AI 预审页采用固定视口骨架：`page-heading` 固定页头、连续 `Statistic` 状态摘要条、单行 `Input.Search / Select / Switch / Segmented` 筛选工具栏、表格或卡片主体内部滚动。状态摘要、筛选栏和主体必须紧贴排列，避免大缝隙、页面级横向滚动和加载后位置跳动；卡片视图只作为同筛选条件下的扫描视图，不重置筛选语义。
- AI 预审任务明细使用独立子页面 `/workspace?page=ai-review-task&task_id=...`，侧边栏仍归属 `AI预审`。详情页结构为返回按钮、任务摘要、覆盖率、提交级明细表格和右侧 `Drawer` 结构化结果；表格展示提交记录、题目序号/短编号、标注员、AI 状态、建议、评分、失败原因、更新时间和操作，完整 job/submission/question 标识仅保留在折叠的“技术信息”中。AI 结果只表达评分、建议和风险，人工审核仍是最终通过/打回入口。
- AI 资源页面强调预算、调用量、成本、模型配置、预警规则；蓝色用于数据趋势，橙色用于预算预警和申请动作。

### 5.6 帮助手册

- `/help` 是终端用户帮助手册，不做营销型落地页，也不承载平台运营或开发者文档。
- `/help` 当前参考 Gitea Docs 的文档站排版和正文格式风格：浅色文档头、左侧章节目录、中间正文文章、右侧本页目录、短段落、步骤列表和 warning/callout；配色继续使用 MarkUp 蓝色主色调，不复制 Gitea 绿色品牌。
- `/help` 正文文案也应参考 Gitea 中文文档的手册写法：每页一个主题，开头短段说明用途，后续用“使用前准备 / 操作步骤 / 注意事项 / 常见情况”类小节组织内容；句子直接说明操作、限制和排查，不写营销式能力介绍。
- `/help` 文字内容必须同步当前已开发功能和 `docs/MarkUp-说明文档/`。调整排版时不得保留与现行任务状态机、企业内流转、数据集/模板版本、AI 预审、人工审核、导出或企业资源不一致的旧说明；建设中能力只能作为限制或后续增强提示出现。
- 首屏应直接提供手册标题、全文搜索和常用入口；搜索、移动端角色筛选、文档分页目录、章节导航和页底翻页使用 Ant Design `Input.Search`、`Segmented`、`Anchor`、`Tag`、`Alert`、`Collapse`、`Button` 和 `Empty`。
- 首屏不展示额外受众说明、布局说明、公开知识源说明、搜索辅助提示或阅读建议提示框；常用入口卡只保留入口标题、图标和目标模块标识。
- 内容按用户路径企业：快速开始、账号与登录、标注员手册、企业/Owner 手册、审核员手册、企业资源、常见问题与故障排查。
- “建设中”能力必须用状态标签标明，并放在相关章节末尾或 FAQ 中，不得写成当前可操作步骤。
- 桌面端采用左侧文档分页目录 + 当前页正文 + 右侧本页目录的工具型文档布局，左侧栏不再放置角色筛选块；中等宽度降为左侧目录 + 正文；移动端改为单列内容和顶部筛选，避免目录挤占阅读空间。正文一次只展示一个章节或 FAQ 页，底部只提供上一页和下一页，不展示数字页码条；上一页/下一页标题必须完整显示，不能省略或挤成多行。不允许把所有手册章节堆成一个长滚动页面。
- 正文行宽需要控制在约 78ch，长内容拆成短段落、步骤列表和普通列表；常用入口卡只做导航，不做营销卖点卡片墙。

## 6. 用户账号管理页面

用户账号管理页面可以参考 Gitee 账号管理页面的信息企业方式：账号相关功能集中在个人设置中，按资料、安全、绑定、通知等分组企业。当前实现按身份拆为两套页面：企业用户使用 Ant Design `Tabs + Card + Form + Descriptions` 的个人账号中心；Labeler 使用独立的基础信息、资质认证、积分管理页面。企业信息、人员管理、资源配置不再混入账号页。

建议导航分组：

| 身份 | 当前分组 |
| --- | --- |
| 企业用户 | 账号概览、基本资料、账号安全、第三方账号 |
| Labeler | 基础信息、资质认证、积分管理 |

账号管理页面设计约束：

- 顶栏登录后在头像左侧提供独立 `工作台` 主按钮；pending 用户进入 `/onboarding`，Platform Admin 进入 `/platform`，其他用户进入 `/workspace`。Platform Admin 手动访问 `/workspace` 时也必须重定向回 `/platform`，避免落入个人或企业工作台兜底视角。
- 头像下拉使用 `Dropdown + Avatar + Descriptions + Tag + Button` 组成身份卡。Labeler 展示个人资料、邮箱、全局角色、积分和资质摘要；企业用户打开下拉时加载默认企业和当前成员身份，展示企业名、企业身份、权限数和角色快捷入口。
- 企业账号页当前分组为账号概览、基本资料、账号安全、第三方账号；不得出现企业资料维护、人员表、资源配置、管理员注册向导、通知偏好或单独的企业与角色子页。
- 企业账号页的 `账号概览` 只展示当前登录人的账号摘要、登录与验证状态、默认企业关系，不展示积分、资质、学历摘要或领域标签等 Labeler 语义信息。
- 企业账号页的 `账号概览` 使用“身份头部卡 + 摘要信息卡”结构，默认企业、企业身份和权限摘要合并到概览内展示，不再拆出独立企业关系标签页。
- 企业账号页的 `基本资料` 只维护企业个人资料，如头像、显示名、真实姓名、手机号、职位/岗位、所在地和个人简介；不维护学历、领域标签等标注员画像字段。
- 企业账号页避免使用大面积灰底信息块；内容区应直接占满工作台主画布，Tabs 导航横向铺满，可编辑表单优先使用更满的双列布局。
- 企业账号页的 `第三方账号` 应使用真实品牌图标，优先使用本地静态 SVG 资源，避免用抽象占位图标代替 GitHub、Google、Hugging Face 的品牌识别。
- 企业账号页的 `账号安全` 中可保留“退出全部其他会话”，但只有在后端能够明确识别当前 refresh session 时才允许执行；若链路未闭环，不得以伪完成状态暴露给用户。
- Labeler 账号页当前分组为基础信息、资质认证、积分管理；基础信息采用 Xpert 风格的学历资料填写，资质认证采用职业认证表单与规则/协议说明页，积分管理只展示收益数据概览并提供等级规则跳转，不复用企业 OAuth/企业资源配置界面。
- 右侧内容按 section 分块，每块有标题、简短说明和表单内容。
- 保存按钮放在每个 section 末尾或页面底部固定操作栏，避免多个无上下文按钮。
- 危险操作如解绑唯一登录方式、注销账号、退出企业必须使用二次确认，并明确不可逆影响。
- 安全状态用标签和说明组合，例如“已验证”“未绑定”“建议开启”。

## 7. 组件规范

| 组件 | 风格要求 |
| --- | --- |
| Button | 优先使用 Ant Design `Button`；主按钮走 `type="primary"`，危险操作用 `danger`，避免自绘按钮体系。 |
| Input / Select | 优先使用 Ant Design `Input`、`Select`、`Form`；保留清晰 label、校验状态和字段级错误。 |
| Card | 优先使用 Ant Design `Card` 或轻量容器；只用于指标、列表项、表单分组，不把页面 section 全部做成浮卡。 |
| Table | 优先使用 Ant Design `Table`，表头、分页、筛选、空态和行操作遵循 antd 交互；列表页表头固定、body 内部滚动、分页固定在右下角。 |
| Tag | 优先使用 Ant Design `Tag`，状态标签保持语义色一致。 |
| Tabs | 优先使用 Ant Design `Tabs`，用于同级视图切换，如任务详情中的题目/提交/审核/统计。 |
| Drawer | 优先使用 Ant Design `Drawer`，用于详情预览、审核操作、成员编辑，宽度建议 420-720px。 |
| Modal | 优先使用 Ant Design `Modal`，只用于确认、短表单、危险操作，不承载复杂工作台。 |
| Empty State | 优先使用 Ant Design `Empty` 或简短空态，包含标题、说明和一个主操作。 |

- 平台级对话 AI 的悬浮态属于固定尺寸工作窗，不允许随消息内容自动撑高；拖拽、缩放和消息滚动必须彼此独立，消息列表单独滚动，输入区和工具栏不参与撑开窗口高度。

加载状态约束：

- 工作台页面级首次加载统一使用基于 Ant Design `Spin` 的 `WorkspaceLoading`，保留稳定高度并通过 `tip` 说明加载对象。
- 表格刷新继续使用 Ant Design `Table loading`，按钮异步操作使用 `Button loading`，卡片骨架可使用 `Card loading`。
- 抽屉和详情区加载使用 `Spin spinning` 或局部 `Spin`；“请选择一条记录”等非加载空态使用 `Empty` 或 `Alert`。
- 不再用裸 `<p>` 或普通文本直接展示“正在加载...”作为工作台加载态。

Ant Design 表格固定区规范：

- 列表页表格统一设置稳定页码区：`pagination.placement = ['bottomEnd']`，启用 `showQuickJumper`，并按业务需要启用 `showSizeChanger`。
- 表格使用固定 body 高度或由页面容器计算剩余高度，数据行在 `Table` 内部滚动；页面标题、概览和筛选栏不得随表格滚动。
- 工作台表格外壳只负责边框、圆角和剩余高度，不得再通过左右 `padding` 制造内缩；固定表格的滚动条占位由 Ant Design Table 自身处理，避免左侧空白条、右侧滚动条空洞和操作列位置抖动。
- 表格容器必须填满筛选条下方到浏览器底部的剩余高度；分页器锚定在表格容器右下角，即使当前数据为空或少于一页，也必须保留每页条数选择和快速跳转入口。
- 表格主列必须设置明确宽度或可截断策略，长文本使用 `ellipsis`、二级说明或详情 Drawer 展示；不把一整段说明塞进主表导致横向抖动。
- 行操作最多保留 2-3 个高频按钮，其余进入 `Dropdown`；危险操作使用 `Popconfirm` 或 `Modal.confirm`。
- 工作台所有 Ant Design 表格的操作列必须对齐模板搭建列表口径：优先复用 `WorkspaceTableActions`，可见动作使用 `Tooltip + icon-only Button`，右侧低频动作进入 `MoreOutlined` Dropdown；常规操作列宽度稳定在约 138px，单动作列可收窄到约 92px，固定右侧列需设置 `workspace-table-action-cell`，避免文字按钮撑宽表格或被滚动容器裁切。
- 过滤条件过多时，主筛选栏只保留关键词和 2-3 个高频筛选，其余放入高级筛选 Drawer，避免筛选栏换行挤压表格主体。

Ant Design 卡片列表规范：

- 生产类卡片视图用于快速扫描，与任务广场任务卡片保持同一信息骨架：顶部状态/分类标签，主体标题和两行描述，中部 3 项关键指标，底部标签与操作区。
- 卡片必须使用 Ant Design `Card`、`Tag`、`Progress`、`Checkbox`、`Dropdown`、`Pagination` 等组件，不自绘卡片控件体系。任务管理卡片支持 `Checkbox` 多选并复用批量操作栏；模板和数据集卡片保留修改、预览/导出、更多操作。
- 卡片区必须替代表格主体容器并填满筛选栏下方到浏览器底部的剩余高度；卡片网格只在内部滚动，底部固定 Ant Design `Pagination`，启用每页条数选择和快速跳转。
- 卡片宽度使用响应式网格，最小宽度约 280-320px；卡片内容采用截断和最多两行描述，禁止因为长标题、长标签或操作按钮撑出页面横向滚动。
- 表格视图和卡片视图共享同一组搜索筛选条件、加载态、空态和业务操作。切换视图不得重置当前筛选语义，只能重置卡片分页到第一页。

## 8. 字体与间距

- 字体：继续使用系统中文字体栈，优先 `PingFang SC`、`Microsoft YaHei`、系统 sans-serif。
- 登录页可保留少量衬线标题以强化品牌感；登录后工作台不使用大面积衬线字体。
- 主标题：20-24px；页面内 section 标题：16-18px；表格和表单正文：14px。
- 间距建议以 4px 为基准：8、12、16、20、24、32。
- 卡片/表单内部间距 16-20px；列表和表格操作区保持紧凑。
- 不使用随视口宽度无限放大的字体；工具型界面优先稳定尺寸。

## 9. 文案风格

- 操作文案使用动词 + 对象：创建任务、导入题目、提交审核、导出数据。
- 状态文案短而具体：待审核、AI 预审异常、已打回、待补充邮箱。
- 错误提示说明可恢复动作：验证码已过期，请重新发送；没有企业权限，请联系企业管理员。
- 不在页面中放大段功能介绍；功能说明放在 tooltip、帮助链接、空状态或文档中。
- 业务操作页不解释系统内部机制、后续接入计划或后台流转细节；页面内只保留必要操作提示、材料要求、风险提醒和校验反馈。
- 工作台主界面不得把 Mongo/ObjectId、job_id、request_id、submission_id、question_id 等长内部标识作为主要阅读对象。列表、卡片、详情头部和筛选文案优先使用任务名、数据集名、模板名、成员名、题目序号、提交记录、状态、时间和业务描述；确需展示内部标识时使用短编号作为二级元信息，并把完整值收进 Drawer/Collapse 的“技术信息”或导出字段配置。

## 10. 响应式要求

- 桌面端优先覆盖 1280px、1440px、1920px 宽度。
- 平板端侧边栏可收起为图标栏；筛选条件改为折叠面板。
- 移动端保留核心查看和轻量操作能力，复杂模板搭建、批量审核、导入导出可提示到桌面端完成。
- 任何按钮、标签、表格单元格都不能出现文字溢出覆盖；长内容使用截断 + tooltip 或换行策略。

## 11. 全站表格增强基线

- `apps/web/src/components/ui/EnhancedTable.tsx` 是当前全站 Ant Design 表格增强层，业务页优先复用，不再在单页重复实现列宽拖拽。
- 默认保留 Ant Design 原生 `Table` 交互语义，只增强高频后台体验：列宽拖拽、横向滚动兜底、固定布局、排序提示。
- 默认策略如下：
  - 开启列宽拖拽，优先通过表头原生拖拽调整列宽。
  - `scroll.x` 默认回退为 `max-content`，避免高密度列在窄容器下直接挤坏。
  - `tableLayout` 默认使用 `fixed`，保证列宽、截断和 sticky 体验稳定。
  - 带 `workspace-fixed-table` class 的表格默认启用 sticky header。
  - 默认启用 `showSorterTooltip={{ target: 'sorter-icon' }}`，减少标题区域提示噪音。
- 表格交互优先级：
  - 优先使用 Ant Design 原生 `filters`、`filterSearch`、`sorter`、`ellipsis`、`pagination`。
  - 低频动作收进 `Dropdown`，不要在每行堆叠过多按钮。
  - 页面级搜索只承接跨列关键词和时间范围；枚举筛选尽量下沉到列级过滤器。
- 若页面需要自定义 `components.header.cell`，必须与增强层兼容，不能绕开共享表格层直接复制一套局部实现。

认证页移动端补充要求：

- 认证页在 320px 宽度起必须可用，登录、注册、忘记密码、OAuth 补绑邮箱流程不能出现横向滚动。
- 桌面端注册表单可使用两列紧凑布局；移动端必须降为单列，邮箱输入 + 验证码按钮也应改为上下排列。
- 移动端首屏优先展示品牌、核心说明和认证卡片，能力指标可以压缩为单列短卡片。
- 所有触控目标高度不低于 44px，协议勾选、忘记密码、返回登录等文本操作必须有足够点击区域。
- 移动端使用 `100dvh` 或等效方案处理浏览器地址栏高度变化，避免认证卡片底部按钮被裁切。
- 用户协议与隐私政策从注册页内打开弹窗阅读；弹窗正文可滚动，关闭按钮放在正文底部，避免用户未进入内容区就误关闭。

## 11. 认证页 Ant Design 约束

认证页继续保留 MarkUp 的品牌入口气质，但交互原语统一收敛到 Ant Design，避免原生表单、自绘按钮和局部自定义弹层并存造成体验割裂。

- 布局采用稳定双栏：桌面端为品牌介绍区 + 认证工作区，推荐 60/40 左右比例；移动端降为单列，品牌摘要在上、认证卡片紧随其后。
- 认证工作区优先使用 Ant Design `Form`、`Form.Item`、`Input`、`Input.Password`、`Checkbox`、`Button`、`Tabs`、`Alert`、`Steps`、`Modal`、`Space.Compact`。
- 登录、注册、忘记密码、OAuth 补绑邮箱应复用同一张认证卡片心智，不再分别做成多个页面风格。
- OAuth 首登 callback 属于认证体系内的特殊弹层任务态，应复用登录/注册同一张认证卡片心智，不再单独做成独立页面或向导页。
- 模式切换使用 `Tabs`，不能再用两颗普通按钮模拟 tab；邮箱 + 验证码发送按钮统一使用 `Space.Compact`。
- 字段反馈以 `Form.Item` 级错误为主，顶部 `Alert` 只承载接口级失败、验证码发送结果、登录成功或流程说明。
- 注册流程要明确提示 pending 用户后续会进入 `/onboarding`；该说明可通过紧凑 `Steps` 或同等级说明区呈现。
- 用户协议和隐私政策继续使用 Ant Design `Modal`，正文可滚动，页脚操作统一为 antd 按钮，不额外套自绘遮罩层。
- 动效控制在 150-300ms，并尊重 `prefers-reduced-motion`；禁止使用会导致布局抖动的 hover 缩放。
- 移动端必须从 320px 宽度起可用，验证码组合、Tabs、按钮和说明卡不得造成横向滚动。
- OAuth 首登 callback 优先使用 Ant Design `Modal`、`Tabs`、`Form`、`Input`、`Alert` 企业为简洁弹层；禁止步骤流、选择向导、多层信息卡和嵌套灰底盒子。

## 14. 企业工作台页头基线

- 企业工作台页头统一复用 `.page-heading`，标题区、动作区、色带和高度不再由各页分别定义。
- 页头主文案与副文案必须垂直居中，避免标题文字偏上；标题区默认使用固定高度和稳定内边距，操作区使用同一套 `page-actions / page-heading-actions` 对齐规则。
- 页头颜色按页面分组统一，不再按单页自由换色：
- `数据生产`：数据集、模板、任务发布/管理等页面使用同一组蓝色系。
- `审核质检`：AI 预审、人工审核使用同一组暖色系。
- `企业管理`：工作台首页、企业信息、资源配置、人员管理、公告通知、操作日志、企业账号页使用同一组治理色。
- Dashboard 和审核页虽然保留各自内容布局，但页头视觉基线必须回到同一套企业工作台体系，不能再去掉色带或改成另一种标题模型。
