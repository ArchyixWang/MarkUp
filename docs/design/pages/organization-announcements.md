# 公告通知页面设计稿

## 1. 页面定位

公告通知承载系统公告、企业通知、审核提醒和导出完成提醒。它不是简单消息列表，而是围绕企业内人员结构、角色和任务关系进行通知分发和追踪的企业级消息中心。

核心目标：

- 查看系统公告和企业通知。
- 管理企业内通知的创建、定向分发和阅读状态。
- 接收审核提醒、打回提醒、AI 预审异常、导出完成提醒。
- 根据人员管理结构按角色、成员、任务、审核队列和企业范围分发消息。
- 支持 WebSocket 实时通知和历史消息查询。

## 1.1 高频使用场景

公告通知页主要服务四类路径：

1. Team Admin / Owner 按角色、成员或任务相关人员发送企业通知，并在发送前确认接收人预览。
2. Reviewer 查看待审核、复审、AI 预审异常等提醒，直接跳转到审核工作台处理。
3. Owner 或导出创建人收到导出完成、失败或下载即将过期提醒，进入导出中心查看文件。
4. 普通成员查看系统公告和企业通知，管理自己的已读、已处理和个人通知偏好。

页面重点是“通知分发和处理追踪”，不是即时聊天。列表、筛选、接收人构建器和详情 Drawer 的优先级高于消息流动效。

## 1.2 消息来源与企业结构

公告通知页必须围绕企业内人员管理结构构建分发，不使用全局用户角色直接判断接收范围。

| 消息来源 | 触发者 | 接收范围 |
| --- | --- | --- |
| 系统公告 | Platform Admin / 系统。 | 全站或指定企业，企业成员只读。 |
| 企业通知 | Team Admin / Owner。 | 全企业、角色、成员、任务相关人员。 |
| 审核提醒 | 审核工作流 / AI 预审工作流。 | assigned reviewer、任务 Owner、相关 Labeler。 |
| 导出提醒 | 导出工作流。 | export creator、任务 Owner、相关企业管理员。 |

当前自动提醒未完全接入时，页面只能展示后端真实返回的通知；不能由前端根据任务或导出状态临时拼出“提醒”。

## 3. 页面结构

```text
公告通知
├─ 固定页面标题
├─ 消息概览状态栏
├─ 通知工具条
├─ Tabs
│  ├─ 全部消息
│  ├─ 系统公告
│  ├─ 企业通知
│  ├─ 审核提醒
│  └─ 导出提醒
├─ 消息表格
├─ 消息详情抽屉
├─ 新建企业通知 Modal
└─ 分发规则抽屉
```

页面采用“表格主视图 + 详情 Drawer + 发送 Modal”的结构。不要做聊天气泡、动态流或卡片瀑布流，因为该页面的核心是筛选、追踪、批量处理和企业级分发。

## 4. 页面头部

左侧：

- 标题：`公告通知`
- 副文案：`管理系统公告、企业通知、审核提醒和导出完成提醒。`

右侧：

- `新建企业通知`，仅 Owner / Team Admin 可见。
- `全部标为已读`
- `通知设置`

## 5. 消息概览

状态栏指标：

| 指标 | 说明 |
| --- | --- |
| 未读消息 | 当前用户未读数。 |
| 企业通知 | 企业内公告数量。 |
| 审核提醒 | 待处理审核、打回、复审提醒。 |
| 导出提醒 | 导出完成或失败提醒。 |
| 系统公告 | 平台级公告。 |

如果后端暂未返回聚合数，前端显示当前筛选内统计。

状态栏交互：

- 点击未读消息切换到全部消息并筛选 `unread`。
- 点击审核提醒切换到审核提醒 Tab，并筛选待处理。
- 点击导出提醒切换到导出提醒 Tab。
- 点击企业通知进入企业通知 Tab；有权限用户可直接新建企业通知。

## 6. 页面状态

### 6.1 加载态

- 消息列表加载使用 `Table` skeleton 或 `Spin`，不清空现有筛选条件。
- 切换 Tabs 时保留搜索关键词和时间范围，类型筛选随 Tab 自动变化。
- 新消息通过 WebSocket 到达时，列表顶部出现 `有新消息，点击刷新` 提示，不强制打断当前阅读。

### 6.2 空状态

- 全部消息为空：展示 `暂无通知`。
- 企业通知为空且用户有权限：展示 `暂无企业通知` 和 `新建企业通知`。
- 审核提醒为空：展示 `当前没有待处理审核提醒`。
- 导出提醒为空：展示 `暂无导出完成提醒`。

### 6.3 错误态

- 列表加载失败：保留筛选条，表格区显示错误和 `重试`。
- 标为已读失败：恢复原状态并提示原因。
- 创建通知失败：保留 Modal 内容，展示字段级错误。

## 7. 通知工具条

工具条：

- 搜索：标题、正文、发送人、任务名。
- 类型筛选：系统公告、企业通知、审核提醒、导出提醒。
- 状态筛选：未读、已读、待处理、已处理、已过期。
- 接收对象筛选：全员、Owner、Reviewer、Labeler、指定成员；系统 Agent 不作为人工通知接收人选项。
- 时间范围。
- 排序：最新、未读优先、待处理优先。

批量操作：

- 批量标为已读。
- 批量设为已处理，仅对提醒类消息开放。
- 批量删除或撤回企业通知仅 Team Admin / Owner 可见，且需后端支持。

工具条布局：

- 第一行放搜索、类型、状态、时间范围和刷新。
- 第二行仅在高级筛选展开时显示接收对象、关联实体、发送人和排序。
- 批量操作条只在选择消息后出现，固定在表格上方，展示已选数量和可操作数量。
- Tabs 切换只改变通知类型，不清空关键词和时间范围。

## 8. 消息表格

使用 Ant Design `Table`。

列：

| 列 | 内容 |
| --- | --- |
| 标题 | 标题、摘要、优先级 Tag。 |
| 类型 | 系统公告 / 企业通知 / 审核提醒 / 导出提醒。 |
| 分发对象 | 角色、成员、任务或全企业。 |
| 状态 | 未读 / 已读 / 待处理 / 已处理 / 已过期。 |
| 关联对象 | task、submission、export、team、member。 |
| 发送人 | 系统、企业管理员、Owner。 |
| 时间 | 创建时间、过期时间。 |
| 操作 | 查看、标为已读、处理、更多。 |

未读行使用轻量背景，不用强烈彩色整行。

表格交互：

- 点击标题打开消息详情 Drawer。
- 点击关联对象进入任务、审核、导出或成员详情。
- 未读消息在标题前显示小圆点或 `Badge`。
- 待处理提醒展示明确主操作：`去审核`、`查看导出`、`处理打回`。
- 已过期企业通知置灰，但仍可查看详情。

## 9. 消息类型

### 9.1 系统公告

平台发布：

- 版本更新。
- 维护通知。
- 安全提醒。
- 平台规则变更。

通常只读，不允许企业成员编辑。

### 9.2 企业通知

Owner / Team Admin 创建：

- 企业规则。
- 项目安排。
- 数据交付提醒。
- 审核排班。

分发对象：

- 全企业。
- 按角色。
- 指定成员。
- 指定任务相关成员。
- 指定审核队列成员。

### 9.3 审核提醒

由工作流产生：

- 待审核提交。
- AI 预审异常。
- 打回需修改。
- 复审/终审提醒。
- 批量审核完成。

关联 `/reviews`、`/ai-reviews` 和任务状态。

### 9.4 导出提醒

由导出中心产生：

- 导出任务创建。
- 导出完成。
- 导出失败。
- 下载即将过期。

关联 `/exports` 和 WebSocket `export:{export_id}`。

### 9.5 优先级与状态

优先级：

- 普通。
- 重要。
- 紧急。

状态：

- 未读。
- 已读。
- 待处理。
- 已处理。
- 已过期。

状态规则：

- 系统公告通常只有未读 / 已读。
- 审核提醒和导出提醒可有待处理 / 已处理。
- 企业通知如果设置过期时间，到期后自动进入已过期。

## 10. 新建企业通知 Modal

字段：

| 字段 | 类型 |
| --- | --- |
| 标题 | Input，必填。 |
| 正文 | TextArea / Rich Text。 |
| 优先级 | 普通、重要、紧急。 |
| 分发对象 | 全企业、按角色、指定成员、指定任务。 |
| 角色 | Owner、Reviewer、Labeler。系统 Agent 不作为人工企业通知角色选项。 |
| 指定成员 | 从人员管理成员列表选择，排除系统 Agent、disabled 和 removed 成员。 |
| 关联任务 | 可选。 |
| 过期时间 | 可选。 |
| 是否邮件通知 | Switch。 |
| 是否站内通知 | Switch。 |

发送前展示分发预览：

- 预计接收人数。
- 角色分布。
- 无法接收的成员。

新建企业通知建议使用两步 Modal 或 Drawer：

1. 编辑内容：标题、正文、优先级、关联对象、过期时间、站内/邮件通道。
2. 选择接收人：全企业、按角色、指定成员、指定任务相关人员，并展示实时预览。

当分发对象复杂时，接收人选择器使用右侧 Drawer，避免 Modal 内部过高。成员选择应复用人员管理成员表字段：成员、角色、邮箱、状态、2FA、职位、手机号；状态为 removed/disabled 的成员默认不可选。

当前 API 已支持企业通知创建、分发预览、全部已读、单条已读/处理、撤回、软删除和个人通知偏好保存。系统公告、审核提醒、导出完成提醒的自动生成和企业级通知策略仍为后续接口需求。

类型口径补充：企业治理页的人工通知在 API 层统一写入并返回 `notification_type=organization`，旧数据中的 `team` 会在响应中映射为 `organization`。页面文案仍可显示为“企业通知/企业通知”，但个人信箱不得硬编码该类型 Tab，必须使用 `/notifications/my` 返回的 `type_options` 动态生成；Labeler 个人信箱不会展示企业广播。

Modal 不承载复杂长表格。接收人预览超过 10 人时，只显示角色分布和成员样例，完整接收人列表进入右侧 Drawer；这样可以避免通知正文、分发条件和预览列表互相挤压。

### 10.1 创建与发送流程

```text
填写通知内容 -> 选择分发对象 -> 预览接收人
-> 确认发送 -> 写通知记录 -> 站内消息/WebSocket/可选邮件 -> 写审计日志
```

发送规则：

- 标题和正文必填。
- 分发对象至少选择一种。
- 紧急通知必须展示确认弹窗。
- 邮件通知依赖 SMTP 配置；未启用时只保留站内通知。
- 分发预览返回 0 人时禁止发送。
- 当前前端发送前会强制完成接收人预览；若用户未手动点击 `预览接收人`，提交时会先调用预览接口。预览返回 0 人时阻止发送并保留 Modal 内容。

发送前校验：

- `全企业` 与 `指定成员` 可以组合，但接收人需去重。
- `按角色` 必须至少选择一个企业内人工角色，不包含系统 Agent。
- `指定任务相关成员` 必须填写任务 ID 或从任务选择器选择任务。
- `审核队列成员` 仅用于审核提醒或企业通知的高级分发，必须依赖 reviewer 分配关系。
- 邮件通知开关在 SMTP 未启用时 disabled，并说明“当前仅发送站内通知”。

### 10.2 消息详情 Drawer

详情内容：

- 标题、正文、优先级、类型、状态。
- 发送人、发送时间、过期时间。
- 分发对象和预计接收人范围。
- 当前用户阅读状态。
- 关联业务对象。
- 操作记录：创建、修改、撤回、已读统计。

企业通知发送者可查看阅读统计：

- 已读人数。
- 未读人数。
- 角色分布。
- 指定成员列表。

详情 Drawer 的主操作随消息类型变化：

| 类型 | 主操作 | 规则 |
| --- | --- | --- |
| 系统公告 | 标为已读。 | 普通成员只读，不显示撤回。 |
| 企业通知 | 标为已读 / 撤回 / 删除。 | 撤回和删除仅创建者、Team Admin 或 Owner 可见。 |
| 审核提醒 | 去审核 / 标为已处理。 | 只能跳转到当前用户有权访问的审核对象。 |
| 导出提醒 | 查看导出 / 标为已处理。 | 已失败导出显示失败原因和重试入口跳转。 |

## 11. 通知分发规则

分发必须围绕人员管理结构：

- 按角色分发：根据 `team_members.team_role`。
- 按成员分发：指定 user_id。
- 按任务分发：任务 Owner、Reviewer、Labeler 贡献者；系统 Agent 不作为人工通知收件人。
- 按审核队列分发：assigned reviewer。
- 按导出任务分发：export creator 和相关 Owner。

### 11.1 分发对象构建器

分发对象构建器是公告通知页的核心交互，必须能解释“为什么这些人会收到通知”。

| 分发方式 | 需要的输入 | 预览展示 |
| --- | --- | --- |
| 全企业 | 无。 | 活跃成员总数、角色分布。 |
| 按角色 | `team_role[]`。 | 每个角色人数和成员样例。 |
| 指定成员 | `user_id[]`。 | 选中成员列表、不可接收原因。 |
| 指定任务相关成员 | `task_id` + 关系类型。 | Owner、Reviewer、Agent、Labeler 贡献者人数。 |
| 指定审核队列成员 | `submission_id` 或审核队列筛选。 | assigned reviewer 列表。 |
| 指定导出相关人员 | `export_id`。 | export creator、任务 Owner。 |

预览返回必须包含：

- `total_recipients`
- `role_breakdown`
- `sample_recipients`
- `excluded_recipients`
- `degraded_reason`，当任务相关人员聚合未完成时使用。

如果预览接口失败，不允许发送；如果降级为企业活跃成员，确认弹窗必须明确提示降级原因。

权限：

- 系统公告：Platform Admin。
- 企业通知：Owner / Team Admin。
- 审核提醒、导出提醒：系统自动生成。

### 11.2 和人员管理的关系

通知分发必须复用人员管理页的数据结构：

- 成员搜索使用 `GET /teams/{team_id}/members`。
- 角色筛选使用企业内角色，不使用全局角色。
- 成员状态为 disabled/removed 时不作为默认接收对象。
- 指定任务相关人员时，需要根据任务 Owner、Reviewer、Labeler 贡献者聚合；系统 Agent 不进入人工通知收件人。

若后端暂未提供任务相关人员聚合，前端可显示“指定任务相关成员”入口，但必须明确提示当前暂按企业活跃成员预览和发送，并要求填写任务 ID 作为关联对象。

### 11.3 自动提醒生成规则

审核提醒和导出提醒不是人工新建消息，后续应由业务工作流写入：

| 触发事件 | 接收人 | 关联对象 | 状态 |
| --- | --- | --- | --- |
| Labeler 提交后进入待人工审核 | assigned reviewer / Owner。 | submission、task。 | pending。 |
| AI 预审异常或失败 | task Owner、Agent、assigned reviewer。 | ai_review_job、submission。 | pending。 |
| 人工打回 | Labeler、task Owner。 | submission、review_record。 | pending。 |
| 导出完成 | export creator、task Owner。 | export_job。 | unread。 |
| 导出失败 | export creator、task Owner。 | export_job。 | pending。 |

当前自动写入未完成时，页面只展示接口返回的真实消息；不要用前端根据任务或导出状态临时生成“假消息”。

## 12. 实时与状态

WebSocket 主题：

- `system:notifications`
- `review:{submission_id}`
- `export:{export_id}`
- `task:{task_id}:stats`

前端行为：

- 新消息进入列表顶部。
- 顶栏通知角标更新。
- 点击消息可跳转关联任务、审核或导出详情。
- 消息已读状态同步到服务端。

实时降级：

- WebSocket 不可用时使用轮询或只依赖手动刷新。
- 顶栏角标无法实时更新时，进入页面后刷新未读数。
- 实时消息缺少完整详情时，仅作为提示，详情仍从 REST API 拉取。

实时更新不能改变用户正在编辑的通知草稿。如果用户正在 Modal 中编写企业通知，新消息只更新顶栏角标或显示轻提示，不自动关闭 Modal 或重载页面。

## 13. API 对应关系

当前通知 REST API 已建立，页面应优先使用下列接口。接口都必须带 `Authorization`，企业作用域能力需要传递 `team_id` 查询参数和 `X-Team-ID`。

| 功能 | API |
| --- | --- |
| 消息列表 | `GET /api/v1/notifications?team_id={team_id}` |
| 创建企业通知 | `POST /api/v1/notifications?team_id={team_id}` |
| 分发预览 | `GET /api/v1/notifications/preview?team_id={team_id}` |
| 全部标为已读 | `POST /api/v1/notifications/mark-all-read?team_id={team_id}` |
| 单条已读 / 已处理 | `POST /api/v1/notifications/{notification_id}/state` |
| 撤回企业通知 | `POST /api/v1/notifications/{notification_id}/revoke?team_id={team_id}` |
| 删除企业通知 | `DELETE /api/v1/notifications/{notification_id}?team_id={team_id}`，软删除，审计日志保留 |
| 个人通知偏好 | `GET /api/v1/profile/me` / `PUT /api/v1/profile/me`，读写 `notification_settings` |

已有关联 API：

- `GET /teams/{team_id}/members`
- `/reviews`
- `/ai-reviews`
- `/exports`
- WebSocket `system:notifications`

当前后续缺口：

- 系统公告、审核提醒、导出完成提醒的自动写入。
- 企业级通知策略和强制通知规则。
- WebSocket 实时推送落地。
- 任务相关人员精确聚合。

在这些能力未接入前，页面可展示说明 Drawer 或只读占位；不能伪造系统自动通知、邮件发送状态或精确分发人数。

## 14. API 字段建议

## 15. 当前可落地流程

一期企业通知创建流程：

```text
打开新建企业通知 -> 填写标题/正文/优先级 -> 选择全企业、角色、成员或任务相关人员
-> 调用分发预览 -> 接收人数 > 0 -> 确认发送 -> 列表刷新 -> 当前用户可读/处理
```

细节规则：

- `target_type=task` 时必须填写关联任务 ID；如果后端任务参与者聚合未完成，页面必须提示“当前按企业活跃成员预览和发送”。
- 紧急通知发送前使用 `Modal.confirm`；用户取消后应完全退出发送流程，不留下 loading。
- `站内通知` 是当前主路径；`邮件通知` 仅在 SMTP 和后端发送链路接入后可显示为可用。
- `通知设置` 当前保存用户个人偏好，不改变企业级强制公告、审核提醒和导出提醒的服务端分发策略。
- 已读/已处理是当前用户视角，不代表全体接收人状态。
- 单条或批量已读/已处理成功后，页面列表、详情抽屉和顶部消息概览必须基于更新后的通知集合同步刷新，不允许概览保留旧的未读数量。
- 当前前端已支持表格多选后的批量标为已读和批量设为已处理，复用单条通知状态接口逐条更新；企业通知撤回和删除仍保留逐条确认，避免批量高风险误操作。

### 15.1 通知设置

通知设置使用 Drawer，不跳出公告通知页。设置内容分为个人偏好和企业策略两层：

| 层级 | 当前能力 | 展示规则 |
| --- | --- | --- |
| 个人偏好 | 通过 `/profile/me.notification_settings` 保存站内/邮件偏好和类型偏好。 | 所有用户可维护自己的偏好。 |
| 企业策略 | 后续由企业级通知策略接口承载。 | 未接入前只读显示 `待接入`，不要把个人偏好误认为企业强制策略。 |

个人关闭某类非强制通知，不应阻止系统安全公告、企业强制公告、审核分配和导出失败等必须通知。

## 15.2 前端实现蓝图

公告通知页要围绕“消息分发、处理和追踪”，不要做聊天界面。

### 首屏布局

```text
固定标题栏
├─ 标题、副文案
├─ 新建企业通知 / 全部标为已读 / 通知设置
└─ 消息概览：未读、企业通知、审核提醒、导出提醒、系统公告

主体
├─ Tabs：全部 / 系统公告 / 企业通知 / 审核提醒 / 导出提醒
├─ 工具条：搜索、状态、接收对象、时间范围、排序
├─ 批量操作条：选中后出现
└─ 消息表格 + 详情 Drawer
```

### 新建企业通知流程

```text
点击新建企业通知
-> 填标题、正文、优先级、过期时间和通道
-> 选择接收范围：全企业 / 角色 / 成员 / 任务相关 / 审核队列
-> 调用分发预览
-> 接收人数 > 0 才允许发送
-> 紧急通知二次确认
-> 发送成功后刷新列表和未读摘要
```

分发预览必须说明“为什么这些人会收到通知”：

| 预览信息 | 展示方式 |
| --- | --- |
| 总接收人数 | 主数字，0 人时禁用发送。 |
| 角色分布 | Tag 或小表格。 |
| 成员样例 | 最多展示 10 人，完整列表进 Drawer。 |
| 不可接收成员 | 显示禁用、已移除、无邮箱等原因。 |
| 降级原因 | `Alert warning`，例如任务相关人员聚合未接入。 |

### 消息表格行为

- 未读行只用轻量背景和 `Badge`，不整行大面积高亮。
- 审核提醒行主操作为 `去审核`；导出提醒行主操作为 `查看导出`。
- 企业通知的 `撤回`、`删除` 放入更多菜单，保留逐条确认。
- 批量操作只允许已读/已处理，不做批量撤回或批量删除。
- 用户正在编辑通知时，新消息只更新角标或显示轻提示，不关闭 Modal。

### 通知设置边界

- 个人偏好通过 `/profile/me.notification_settings` 保存。
- 企业强制通知策略未接入时，只读展示 `待接入`。
- SMTP 未启用时，邮件通道 disabled，文案为 `当前仅发送站内通知`。
- 系统公告、审核提醒、导出提醒的自动写入未接入时，不由前端合成假消息。

## 18. 验收标准

- 消息中心以表格为主视图，支持搜索、筛选、排序和批量已读。
- 企业通知创建流程包含分发对象和接收人预览。
- 审核提醒和导出提醒能跳转到对应业务对象。
- 普通成员不能创建或撤回企业公告。
- WebSocket 不可用时页面仍能通过 REST 或手动刷新查看历史消息。

## 19. Hallmark 检查

- 不做聊天界面；这是企业级通知管理表格。
- 不把所有消息做成卡片流，批量处理和筛选必须清晰。
- 不伪造分发人数。
- 不让普通成员创建企业公告。
- 审核和导出提醒必须能跳转业务对象。
