# 人员管理页面设计稿

## 1. 页面定位

人员管理是企业工作台的企业成员治理页面，主要用于查看企业所有成员、搜索筛选成员、管理角色与状态、邀请或添加成员，并对高风险成员操作进行审计。

页面参考 Gitee 企业成员管理类页面的信息企业方式：主页面以成员表格为核心，顶部提供表格名称、搜索、筛选、排序、添加成员和更多操作。MarkUp 不照搬视觉，仍使用当前 Ant Design 工作台风格。

设计原则：

- 主页面就是成员表格，不做卡片墙。
- 管理动作集中在表格上方工具条和行内更多菜单，不把长表单常驻铺在页面下方。
- 添加成员、删除成员、批量操作等高权限动作只对 Owner 或具备成员管理权限的用户显示。
- 权限判断优先使用后端返回的 `permissions` / `actions` 字段，不只依赖角色名称。
- 成员管理必须保留审计语义，关键操作需二次确认。
- `Agent` 为系统 Agent，企业创建时自动存在；人员管理页必须展示该角色，但不能提供创建、邀请、编辑、禁用、删除、批量操作或任务转交入口。

## 2. 入口与命名

- 侧栏条目：`人员管理`。
- 所属分组：`企业管理`。
- 图标：继续使用企业/成员语义图标，例如 Ant Design `TeamOutlined`。
- 点击后进入人员管理主页面。

### 2.2 面包屑

列表页：

```text

```

成员详情抽屉不改变面包屑。

若后续扩展为独立成员详情页：

```text

```

| 成员详情 | `/workspace/people/:userId` | 可选深链接，短期先用 Drawer。 |

## 3. 用户目标

Owner 进入人员管理后的核心目标：

1. 快速查看企业所有成员及其角色、安全状态、联系方式和账号状态。
2. 搜索某个成员，按角色、状态、职位、加入时间等维度筛选。
3. 添加成员或邀请成员加入企业。
4. 修改成员角色、状态、职位、手机号等企业内资料。
5. 查看成员承担的任务、审核队列、权限摘要和最近活跃情况。
6. 移除、禁用或锁定成员，并明确不可逆影响。
7. 导出成员清单或查看操作日志。

## 5. 页面头部

页面标题区固定在面包屑下方。

左侧：

- 标题：`人员管理`
- 副文案：`查看企业成员、角色、安全状态和联系方式，管理成员加入、权限变更与移除。`

右侧：

- 可选状态提示：`当前企业：MarkUp 测试企业`
- 可选按钮：`刷新`

主操作不放在页面标题最右侧，而放在表格工具条中，符合“表格页工具条驱动”的使用习惯。

## 6. 表格工具条

工具条从左到右：

```text
成员列表 | 搜索成员 | 角色筛选 | 状态筛选 | 排序 | 添加成员 | ...
```

### 6.1 表格名称

- 文案：`成员列表`
- 旁边显示当前筛选结果数量，例如 `共 128 人`。
- 如果只显示当前页数量，必须写成 `当前筛选 20 人`，不要伪造总数。

### 6.2 搜索栏

搜索范围：

- 成员姓名 / 昵称。
- 用户名。
- 邮箱。
- 手机号。
- 职位。

当前 API 支持 `keyword`，前端先将搜索值传给 `GET /teams/{team_id}/members?keyword=`。

### 6.3 筛选栏

建议筛选项：

| 筛选 | 选项 |
| --- | --- |
| 角色 | 全部、Team Admin、Owner、Reviewer、Agent、Labeler。 |
| 成员状态 | 全部、正常、禁用、锁定、待邀请、已离开。 |
| 邮箱状态 | 全部、已验证、未验证。 |
| 职位 | 管理员、任务负责人、审核员、标注员、Agent、自定义职位。 |
| 加入时间 | 最近 7 天、30 天、90 天、自定义范围。 |

当前后端未返回职位和手机号时，前端应显示 `-` 或 `未配置`，不伪造状态。

### 6.4 排序

排序选项：

- 最近加入。
- 最近活跃。
- 角色。
- 姓名。
- 任务数。
- 安全风险优先。

当前 API 若暂未支持排序，前端可对当前页本地排序；后续下沉为 `sort` / `order` 参数。

### 6.5 添加成员

- 按钮：`添加成员`，Ant Design `Button type="primary"`。
- 仅 Owner 或具备成员管理权限的用户可见。
- 无权限用户不显示按钮；如果为了解释权限可显示禁用按钮，则必须带 Tooltip：`只有 Owner 可以添加成员`。

点击后打开 `添加成员` Modal，不跳出当前表格。

### 6.6 更多操作

最右侧使用省略号按钮 `...`，仅 Owner 或具备成员管理权限的用户可见。

菜单项：

- `批量导入成员`
- `导出成员清单`
- `发送安全提醒`
- `查看邀请记录`
- `查看成员操作日志`
- `批量删除成员`

危险操作放在菜单底部，使用 `danger` 样式，并且二次确认。

### 6.7 批量导入成员

批量导入从更多菜单打开 Modal，不跳转页面。当前一期使用 CSV 文本粘贴，后续可复用同一接口接入文件上传解析。

输入格式：

```text
email,role,username,display_name,password
labeler3@example.com,labeler,labeler03,Labeler Three,SecurePass123!
```

交互规则：

- 支持表头 `email,role,username,display_name,password` 或中文表头，提交前前端解析为结构化 JSON。
- `role` 输入只支持 `team_admin`、`owner`、`reviewer`、`labeler`，可兼容常见中文角色名；`agent` 为系统 Agent，不接受人工导入。
- 新邮箱会创建账号并加入企业；已有账号且未加入当前企业时直接加入企业。
- 新账号必须提供行内密码或默认初始密码；密码必须符合后端强度规则。
- 已在企业内的成员、导入文本内重复邮箱、用户名冲突、缺少密码等逐行跳过，不阻断其他行。
- 导入完成后展示 `已导入 N 名成员，跳过 M 行`，并刷新成员表格。
- 后端必须写入逐成员审计和企业级批量导入汇总审计。

## 7. 成员表格

使用 Ant Design `Table`。

### 7.1 列设计

| 列名 | 内容 | 交互 |
| --- | --- | --- |
| 成员 | 头像、显示名、用户名、账号状态 | 点击打开成员详情抽屉。 |
| 角色 | 企业角色 Tag、全局角色小字 | Owner 可在编辑抽屉修改。 |
| 职位 | 企业内职位或职能，例如任务负责人、审核员 | 可编辑。 |
| 手机号 | 手机号，未配置显示 `-` | 不完整手机号可脱敏显示。 |
| 邮箱 | 邮箱、验证状态 | 未验证显示 Warning Tag。 |
| 任务/审核 | 负责任务数、分配审核数、AI 资源管理标记 | 悬停显示摘要。 |
| 最近活跃 | 最近登录或最近操作时间 | 无数据显示 `-`。 |
| 加入时间 | 加入企业时间 | 支持排序。 |
| 操作 | 查看、编辑、更多 | 高权限操作收进行内更多。 |

如果列过多，桌面端固定 `成员` 列和 `操作` 列，中间列横向滚动。默认桌面宽度 1440px 下应可读。

### 7.2 成员列

成员列展示：

```text
[头像] 张三
      zhangsan · active
```

状态：

- active：正常。
- disabled：已禁用。
- locked：已锁定。
- invited：待接受邀请。
- removed：已移除，默认不展示，可在筛选中查看。

当前用户行需要标记 `我`，并禁用移除自己。

### 7.3 角色列

角色使用语义 Tag：

- Team Admin：深色或蓝色。
- Owner：橙色。
- Reviewer：紫色或蓝色。
- Agent：青色。
- Labeler：绿色。

角色说明 Tooltip：

- Team Admin：管理企业、预算和成员。
- Owner：管理任务、模板、数据集、导出和成员。
- Reviewer：审核队列和提交结果。
- Agent：AI 资源观察与相关运维协作。
- Agent 行继续展示角色标签，但不额外追加“系统 Agent”或“只读”的显式提示框；不可编辑、不可删除等限制仅通过禁用操作体现。
- Labeler：领取任务和提交标注。

### 7.4 操作列

普通用户可见：

- `查看`

Owner 可见：

- `编辑`
- 行内 `...` 更多菜单：
  - `重置角色`
  - `禁用成员`
  - `发送安全提醒`
  - `重发邀请`
  - `移除成员`

操作规则：

- 不允许移除自己。
- 不允许移除最后一个 Owner / Team Admin。
- 移除 Owner 前，如果仍负责发布中任务，提示先转交任务负责人。
- 禁用成员不删除历史提交、审核和审计记录。

## 8. 添加成员 Modal

点击 `添加成员` 打开 Modal，提供三种方式：

### 8.1 邀请码邀请

- 复用 `POST /teams/{team_id}/invite`，但使用 `invite_mode=code` 生成企业邀请码。
- 管理员只填写成员角色、邀请说明和有效期，不填写受邀邮箱。
- 提交后弹窗内直接展示 `invite_code`、`invite_url` 和过期时间，并提供复制按钮，便于通过 IM、文档或人工渠道转发。
- 明确提示“供已注册用户在 onboarding 填写企业邀请码加入”。
- 该模式不绑定邮箱，接收方只要是已登录的 `pending` 用户即可在 onboarding 填码加入。

### 8.2 邮箱邀请

字段：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| 邮箱 | Input | 必填，邮箱格式。 |
| 角色 | Select | 必填，默认 Labeler 或 Owner 按业务策略确定。 |
| 邀请说明 | TextArea | 可选。 |
| 过期时间 | Select | 24h、72h、7 天。 |

提交调用 `POST /teams/{team_id}/invite`。

### 8.3 创建成员账号

字段：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| 用户名 | Input | 必填，唯一。 |
| 显示名称 | Input | 可选，默认用户名。 |
| 邮箱 | Input | 必填。 |
| 手机号 | Input | 可选，后端支持后写入 profile。 |
| 职位 | Input | 可选。 |
| 初始角色 | Select | 必填。 |
| 初始密码 | Password | 必填，符合强度规则。 |
| 是否邮件通知 | Switch | 默认开启。 |

提交调用 `POST /teams/{team_id}/members/accounts`。

说明：

- 如果后端暂未支持职位、手机号写入成员接口，前端先在设计中保留字段，实际实现时只提交 API 支持字段，并在后续接口需求中补充。
- 密码输入必须支持显示/隐藏。

## 9. 成员详情 Drawer

点击成员列或 `查看` 打开 Drawer。

内容：

- 基本信息：头像、姓名、用户名、邮箱、手机号、职位、加入时间。
- 角色与权限：企业角色、权限数量、权限摘要。
- 安全状态：邮箱验证、账号状态、最近登录。
- 生产关系：负责任务数、审核任务数、AI 调用关联数、提交数。
- 最近操作：关键审计日志。

底部操作：

- Owner：`编辑成员`、`禁用成员`、`移除成员`。
- `查看成员操作日志`：跳转到操作日志页，默认筛选 `entity_type=team_member`、`entity_id=user_id`。
- 普通用户：无管理按钮。

## 10. 编辑成员 Drawer

Owner 点击 `编辑` 打开 Drawer。

字段：

- 企业角色。
- 职位。
- 手机号。
- 成员状态：正常、禁用。
- 审核任务分配范围。
- 权限摘要，只读或高级编辑预留。

保存调用：

- `PUT /teams/{team_id}/members/{user_id}` 更新角色、权限、审核任务。
- 个人资料字段如职位、手机号后续可通过用户/资料接口或新增企业成员 profile 接口补齐。

保存前检查：

- 修改角色会影响访问权限，需提示。
- 禁用成员会阻止登录或企业访问，需确认。
- Reviewer / Owner 有未完成任务时，提示先处理相关任务。

## 11. 删除与禁用

### 11.1 移除成员

移除成员使用 `DELETE /teams/{team_id}/members/{user_id}`。

确认弹窗必须展示：

- 成员姓名。
- 当前角色。
- 负责任务数 / 分配审核数。
- 影响说明：移除后无法访问企业，但历史提交、审核、任务和日志仍保留。

高风险场景：

- 移除自己：禁用。
- 移除最后一个 Owner / Team Admin：禁用。
- 有未完成任务：要求转交或确认跳过。

### 11.2 禁用成员

禁用通过 `PUT /teams/{team_id}/members/{user_id}` 更新 `status=disabled`。

禁用适合临时冻结访问，不删除企业关系。禁用后表格仍展示该成员，可通过状态筛选查看。

## 12. 批量操作

勾选表格行后显示批量操作栏：

- 批量修改角色。
- 批量发送安全提醒。
- 批量禁用。
- 批量移除。
- 批量导出所选成员。

规则：

- 批量操作只对 Owner 可见。
- 当前用户、最后一个 Owner、不可操作成员自动跳过。
- 执行前展示可操作数量和跳过原因。
- 当前前端已支持表格选择后的 `批量改角色`，可编辑成员会进入批量角色 Modal；当前用户和不可编辑成员跳过。提交后调用后端批量角色接口，成功成员重置为目标角色默认权限并写入审计日志。
- 当前前端已支持 `发送安全提醒`：表格多选后可批量向可编辑且 active 的成员发送安全提醒，行内更多菜单可对单个成员发送提醒。提交后调用 `POST /teams/{team_id}/members/security-reminders`，后端创建成员定向企业通知并写入 `member_security_reminder_sent` 审计日志；当前仅发送站内通知，邮件通道由后续企业通知策略统一控制。

## 13. 邀请记录

从更多菜单打开邀请记录 Drawer。

列：

- 邀请邮箱。
- 角色。
- 邀请人。
- 状态：待接受、已接受、已过期、已撤销。
- 过期时间。
- 操作：重发、撤销。

当前 API 已有邀请、邀请记录列表、重发、撤销和响应接口；前端人员管理页从更多菜单打开邀请记录 Drawer，展示邀请邮箱、角色、邀请人、状态、过期时间和响应时间。待接受或已过期邀请可在行内执行 `重发` 与 `撤销`，已接受、已拒绝和已撤销邀请只读展示。

## 14. 权限与安全

权限口径：

- 添加成员、编辑成员、禁用成员、移除成员、批量操作、邀请记录管理：仅 Owner 或后端 `actions/permissions` 明确允许时展示。
- Team Admin 在 API 文档中具备企业管理能力；如果产品口径要求页面上只显示 Owner 操作，应由后端权限或企业角色映射统一返回，前端不硬编码绕过。
- Reviewer、Agent、Labeler 默认只读或不可见管理入口。

安全要求：

- 邮箱未验证时显示风险 Tag。
- 操作日志记录添加、邀请、角色修改、禁用、移除和导出。
- 不在错误提示中暴露敏感信息。

## 15. API 对应关系

| 前端功能 | API |
| --- | --- |
| 企业概览 | `GET /api/v1/teams/admin/overview` |
| 成员列表 | `GET /api/v1/teams/{team_id}/members` |
| 添加已有用户 | `POST /api/v1/teams/{team_id}/members` |
| 创建成员账号 | `POST /api/v1/teams/{team_id}/members/accounts` |
| 邀请成员 | `POST /api/v1/teams/{team_id}/invite` |
| 邀请记录 | `GET /api/v1/teams/{team_id}/invitations` |
| 重发邀请 | `POST /api/v1/teams/{team_id}/invitations/{invitation_id}/resend` |
| 撤销邀请 | `POST /api/v1/teams/{team_id}/invitations/{invitation_id}/revoke` |
| 接受/拒绝邀请 | `POST /api/v1/teams/invitations/{invite_code}/respond` |
| 更新成员 | `PUT /api/v1/teams/{team_id}/members/{user_id}` |
| 批量改角色 | `POST /api/v1/teams/{team_id}/members/batch-role` |
| 移除成员 | `DELETE /api/v1/teams/{team_id}/members/{user_id}` |
| 成员详情 | `GET /api/v1/users/{user_id}` |
| 操作日志 | `GET /api/v1/audit-logs` |

当前前端已接入人员管理到操作日志页的内部状态跳转：顶部更多菜单使用 `entity_type=team_member` 查看成员类审计记录，成员详情抽屉使用 `entity_type=team_member` + `entity_id=user_id` 定位单个成员。后续改为真实路由时，应迁移为 URL 查询参数。

当前前端已支持表格多选后的批量移除和批量改角色：不可移除/不可编辑成员和当前用户在选择框或提交结果中跳过；批量移除复用单成员 `DELETE /teams/{team_id}/members/{user_id}` 逐个执行，批量改角色调用 `POST /teams/{team_id}/members/batch-role` 并返回逐项结果。后续如成员量较大，可补专用批量移除接口以返回逐项成功/失败明细。

后续建议补充接口字段：

- `mfa_enabled` / `two_factor_enabled`
- `phone`
- `position`
- `last_active_at`
- `joined_at`
- `invitation_status`
- `manageable_actions`
- 批量邀请接口和专用批量移除接口。

## 16. Ant Design 组件建议

| 区域 | 组件 |
| --- | --- |
| 工具条 | `Space`、`Typography.Text`、`Input.Search`、`Select`、`Dropdown`、`Button` |
| 成员表格 | `Table`、`Avatar`、`Tag`、`Tooltip`、`Badge` |
| 添加成员 | `Modal`、`Form`、`Input`、`Select`、`Switch` |
| 成员详情 | `Drawer`、`Descriptions`、`Tabs`、`Timeline` |
| 编辑成员 | `Drawer`、`Form`、`Select`、`Input`、`Alert` |
| 危险确认 | `Modal.confirm`、`Popconfirm` |
| 批量操作 | `Alert` 或固定批量操作栏 |

## 17. Hallmark 反模板化检查

本页必须避免：

- 做成多张成员卡片，导致企业成员扫描效率低。
- 把添加成员表单常驻在表格下方，挤占主表格空间。
- Owner-only 操作只靠按钮禁用，不解释权限原因。
- 把删除成员放在显眼主按钮位置；危险操作必须收进更多菜单并二次确认。
- 伪造手机号、职位、最近活跃等后端未返回字段。
- 不区分禁用和移除，导致误删企业关系。
- 不处理“移除自己”“移除最后一个 Owner”“移除仍有任务成员”等高风险约束。

## 18. 实施优先级

第一阶段：

- 独立人员管理页。
- Ant Design Table 主表格。
- 搜索、角色筛选、状态筛选、排序。
- 添加成员 Modal。
- 编辑成员 Drawer。
- Owner-only 操作可见性。

第二阶段：

- 职位、手机号、最近活跃字段接入。
- 行内更多菜单和危险确认。
- 批量操作增强：批量改角色、批量安全提醒和专用批量移除接口。
- 批量邀请。

第三阶段：

- 成员详情深链接。
- 权限细粒度编辑。
- 操作日志列表。
- 任务负责人转交流程与生产任务 ownership 迁移。
