# 认证入口页设计稿

## 页面定位

认证入口页是 MarkUp 的品牌入口与账号流量收口页，既承载首次访问用户的产品认知，也承载已知用户的登录、注册、重置密码和 OAuth 补绑邮箱流程。页面需要同时适配独立路由页和首页登录弹层。

## 布局规则

- 桌面端使用双栏布局，左侧为品牌介绍区，右侧为认证工作区，推荐视觉比例约 60/40。
- 移动端降为单列：品牌摘要在上，认证卡片在下，能力证明卡压缩为 2-3 张短卡，禁止横向滚动。
- `overlayMode` 与独立认证页复用同一套认证卡片内容，只允许壳层容器不同。

## 品牌介绍区

- 展示产品名称、一句话价值主张、3 条能力摘要和 3 张能力证明卡。
- 语气偏“数据生产工作台”，不要写成营销落地页大段文案。
- 背景可保留浅色渐变与轻装饰层次，但应克制，避免登录后工作台风格断层。

## 认证工作区

- 使用 Ant Design `Form` 体系承载所有输入和校验。
- 头部层级固定为：标题、副标题、模式切换 Tabs、反馈区、表单区、辅助入口。
- 登录和注册使用 `Tabs` 切换。
- 忘记密码和 OAuth 补绑邮箱作为同一卡片中的任务态切换，不再另起一套页面心智。

## 字段与反馈规范

- 登录字段：邮箱/用户名、密码、保持登录状态、忘记密码入口。
- 注册字段：用户名、账号、邮箱、邮箱验证码、密码、协议勾选。
- 注册表单需明确提示“创建账号 -> 完成身份分流”，说明 pending 用户登录后进入 `/onboarding`。
- 邮箱 + 验证码发送按钮统一使用 `Space.Compact`。
- 密码输入统一使用 `Input.Password`，依赖组件原生显隐能力。
- 字段错误优先展示在 `Form.Item` 下方；顶部 `Alert` 只展示接口级错误、发送结果、成功提示和流程说明。

## 协议与弹窗

- 用户协议与隐私政策从注册区内打开 Ant Design `Modal`。
- 弹窗正文滚动，页脚提供单一关闭动作。
- 法律正文沿用当前仓库文案，不在本页设计稿中改写内容。

## 交互约束

- 验证码发送成功后显示轻量说明并启动 60 秒倒计时。
- 倒计时期间发送按钮禁用，但表单提交流程仍可继续。
- 除加载态与倒计时外，不依赖“按钮预禁用”替代表单校验。
- 登录成功后 pending 用户进入 `/onboarding`，Platform Admin 进入 `/platform`，其他非 pending 用户进入 `/workspace`。
- 会话失效重新唤起登录时，认证卡片应直接可用，不能停留在“工作台已开但内部提示请先登录”的割裂状态。

## OAuth 首登账号接入

- `/oauth/callback` 继续保留为前端回调入口，但视觉上不再使用独立页面。
- callback 命中后由当前路由承载首页背景，并自动打开与登录/注册同一套 Ant Design 认证弹层。
- OAuth 首登不再使用步骤流、选择卡或多层信息卡；内部直接复用单卡认证结构。
- 关联方式使用 `Tabs` 切换：
  - `绑定已有账号`
  - `注册新账号`
- 绑定已有账号与注册新账号切换时，弹层主卡高度保持稳定，避免内容高度跳变影响体感。
- 若 provider 已返回可信邮箱，在注册态中以只读输入展示该邮箱；仅在 provider 未返回可信邮箱时，才展示邮箱 + 验证码输入组合。
- 第三方可信邮箱仅作为展示或注册默认值，不要求与现有 MarkUp 账号邮箱一致。
- 用户关闭 OAuth 关联弹层时，回到普通登录弹层，不继续停留在 callback 专属流程。
- 第三方登录入口使用 Ant Design 图标按钮，当前 provider 为 GitHub、Google、Hugging Face；避免文字按钮占据过多空间。

## 响应式约束

- 320px 起可用，触控目标高度不低于 44px。
- 小屏下 Tabs、验证码组合、协议勾选与主按钮全部换成单列或自然换行布局。
- 认证卡片底部主按钮不可被浏览器地址栏遮挡，必要时使用 `100dvh` 或等效策略。
