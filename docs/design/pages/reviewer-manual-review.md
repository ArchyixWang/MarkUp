# Reviewer 人工审核页设计稿

## 1. 页面定位

人工审核是 Reviewer 工作台的核心生产页，承担 AI 预审结果复核、人工终审、批量处理、历史追溯和账号概览等工作。它不是任务广场，也不是配置页，而是一个面向高密度审查与快速处置的三栏审阅工作台。

设计基线：

- 组件统一使用 Ant Design。
- 前端视觉必须沿用已完成的任务管理页风格：浅蓝模块主色、固定页头、紧贴统计条、单行筛选操作条和固定视口内部滚动，不另起一套审核页视觉体系。
- 页面固定在一个浏览器视口内，不依赖浏览器页面上下滚动承载主流程。
- 左、中、右三栏分别承载队列管理、审核详情和账号概览，三栏内部独立滚动。
- 参考 `docs/assets/验收.png` 的信息分区方式，但视觉收敛到 MarkUp 当前浅蓝白灰工作台风格。
- 审核动作、批量动作和历史追溯必须显式，不能只靠提示文案隐藏流程。

## 2. 入口与侧栏

### 2.1 工作台侧栏

Reviewer 工作台建议分为四个分组：

```text
主页面
├─ 主页面

审核质检
├─ AI预审
├─ 人工审核

企业管理
├─ 企业信息
├─ 资源配置
├─ 人员管理
├─ 公告通知
└─ 工作日志

个人工具
└─ 账号管理
```

说明：

- `AI预审` 供 Reviewer 查看任务级 AI 预审覆盖率、失败任务、提交级评分和 AI 建议；`人工审核` 是 Reviewer 进入后的最终人工复核入口。
- `工作日志` 作为 Reviewer 可见的企业治理入口之一，命名沿用前端可读文案，后续可映射到操作日志模块。
- Reviewer 不应看到 Owner 的生产配置入口，也不应看到无关的任务创建向导。

### 2.2 面包屑

```text
工作台 / 人工审核
```

若后续拆分到二级详情页，建议保持父级面包屑稳定，不要在队列切换或详情切换时反复闪烁。

## 3. 页面目标

Reviewer 进入人工审核页后的核心目标：

1. 快速识别 AI 建议通过、AI 建议打回和待人工审核三类队列。
2. 批量通过、批量打回、批量指派。
3. 查看单条提交的原始数据、标注内容、AI 评语、评分维度、历史轮次和审计轨迹。
4. 在中间详情区完成通过、打回、直接修订和入库等最终动作。
5. 在右侧快速确认个人工作量、待办量、通过率和当日处理情况。

## 4. 信息架构

```text
人工审核页
├─ 固定页头
├─ 三栏工作区
│  ├─ 左侧：AI 预审结果队列
│  ├─ 中间：审核详情与操作区
│  └─ 右侧：账号与工作概览
├─ 批量操作确认弹窗
├─ 打回/修订原因弹窗
├─ 指派/转交弹窗
└─ 审计时间线 / 历史 Drawer
```

页面主体不允许出现浏览器整页滚动。所有长内容都应进入各自区域的内部滚动容器，避免标题栏、筛选栏、右侧概览栏在滚动中丢失上下文。

## 5. 页面头部

页面标题区固定在面包屑下方，顶部对齐当前工作台内容区。

左侧：

- 标题：`人工审核`
- 副文案：`复核 AI 预审结果、批量处置提交内容、查看历史轮次和审计轨迹。`

右侧：

- `刷新`，刷新当前统计、队列和选中详情。
- `导出审计日志`，用于导出当前筛选范围内的审核轨迹。
- `收起标题栏` 放在搜索与批量处理工具栏最右侧。收起后标题栏完全隐藏，不再生成小浮框或额外控制条；同一按钮切换为 `展开标题栏`，避免页面向下移动或正文被浮层挤压。

标题栏不再提供 `全部阶段 / 初审 / 复审 / 终审` 的独立切换控件。Reviewer 的真实切换入口改为右侧“当前提交审计时间线”：同一轮次内的标注员提交、AI Agent 预审、人工复核、打回/通过等日志用浅色虚线框分组。点击第一轮分组进入第一轮初审视角；点击第二轮分组进入第二轮复审视角；后续更多轮次继续按轮次分组点击切换。分组内不显示解释性提示小字，每条日志采用 `账号名 + 时间` 作为首行，下方显示操作摘要和必要备注，参考验收示意图右侧提交时间线的信息密度。该设计把视角切换绑定到真实审计证据，避免标题栏阶段按钮和审计轮次脱节。

页头必须紧贴主工作区，不做大面积 hero 式留白。

## 6. 三栏工作区

### 6.1 整体结构

三栏建议比例：

- 左栏：320-360px，偏窄，承载队列和批量操作。
- 中栏：弹性占满主体，是主要审核工作区。
- 右栏：280-340px，承载个人概览和快速状态。

三栏之间保持稳定间距，禁止页面横向抖动。左右栏可内部滚动，但页面本身不应出现左右拖动。

### 6.2 左侧队列栏

左栏是 AI 预审结果管理区，顶部必须先给出三类状态切换，再提供筛选和批量动作。

#### 顶部筛选

建议使用 Ant Design `Segmented` 或紧凑 `Tabs`：

- `AI建议通过`
- `AI建议打回`
- `待人工审核`

当前实现不在筛选条重复放置 `AI建议通过 / AI建议打回 / 待人工审核` 三个按钮，避免和统计条重复。统计条继续沿用任务管理页 `WorkspaceSummaryStrip` / Ant Design `Statistic` 风格，只收窄宽度并支持点击；`待审核`、`AI建议通过`、`AI建议打回`、`待人工审核` 和 `已处理` 点击后刷新左侧队列。统计条只显示标题和数字，不显示“来自当前权限范围”等辅助小字。

筛选补充项建议：

- 任务
- 数据集
- 标注员
- 审核轮次
- AI 分数区间
- 风险标签
- 时间范围
- Assigned only

#### 批量操作栏

勾选多条后，在列表顶部显示批量操作条，使用 Ant Design `Checkbox`、`Button`、`Dropdown` 组合。

支持操作：

- 批量通过
- 批量打回
- 批量转交 / 指派
- 批量标记为待复核
- 批量导出当前队列

批量打回和批量转交必须二次确认或填写原因。批量动作的结果要明确显示成功、失败和跳过数量。

搜索栏与处理按钮合并为一行，右侧直接展示 `批量通过`、`批量打回`、`指派给其他 Reviewer` 三个按钮。提交级指派接口未接入前，指派按钮打开说明弹窗，不伪造实际转派结果。

#### 队列条目

每条队列建议展示：

- Submission 编号或数据集行号
- 任务名 / 模板名
- 标注员信息
- 提交时间
- 标题
- 概要
- AI 得分
- AI 建议
- 轮次标签
- 风险标签
- 当前状态

队列条目交互：

- 单击选择后，在中间区加载对应详情。
- 当前选中项高亮。
- 风险标签和 AI 建议使用语义色 Tag，但不要堆满装饰色块。

### 6.3 中间审核详情栏

中间栏是主要审核工作区，分为内容阅读、AI 结果、审稿意见和最终动作四层。

#### 顶部信息

建议显示：

- Submission 编号
- 任务标题
- 轮次
- 当前状态
- 数据集或模板名

#### 原始内容与标注内容

优先采用双块或上下对照布局：

- 原始数据
- 标注结果 / 提交答案
- 关键字段高亮
- 变更字段 Diff

如果是结构化 JSON，可使用紧凑的 Key-Value 展示或代码块，但必须可读。

#### AI 预审结果

AI 结果区域应单独高亮，使用浅蓝底或浅蓝边框，包含：

- Provider 名称与对应模型，优先从任务 `ai_config.provider_id` 关联企业 AI Provider 配置解析，失败时显示 Provider ID 或未配置状态。
- 总分
- 各维度评分
- AI 建议
- 风险提示
- 置信度 / 版本标记（如有）

AI 预审摘要采用紧凑指标网格，不再使用大面积统计卡堆叠。首行展示 Provider、模型、AI job 状态、总分、风险数量和轮次；AI 原因使用单行浅蓝信息块承载，维度评分继续使用 Ant Design `Table` 小尺寸展示，确保中间详情栏一屏内优先看到评分、差异和答案主体。如果当前提交没有 AI 预审结果，只显示一行浅蓝提示“无 AI 预审”，不得保留大面积空卡片或空矩形。

如果有多维度评分矩阵，应展开显示每个维度的名称、分值、简述和风险原因。

详情区采用“一图流”纵向审阅：顶部显示 AI 总分、建议、风险和维度评分；中段用表格展示当前轮次对应的字段差异，第一轮只称为“第一轮提交字段差异”，第二轮及以后按 `第 N 轮字段差异` 展示，标题下方不再放说明性副文案，避免占用审阅空间；底部并排展示原始数据和当前标注答案 JSON。当前审核视角由右侧审计时间线的轮次分组点击决定，详情标题同步显示 `第一轮初审视角`、`第二轮复审视角` 或后续 `第 N 轮审核视角`。

#### 历史与 Diff

建议提供：

- 第 1 轮 / 第 2 轮 / 当前轮对比
- 审核历史时间线
- AI 评语与人工评语对照
- 修改前后内容 Diff

历史信息默认放在右侧栏的当前提交审计时间线中，随左侧选中项更新；中间栏优先保留 AI、差异和答案内容。

#### 最终动作区

最终动作固定在详情区底部，保持易触达：

- `打回`
- `直接修订`
- `通过入库`
- `下一条`

危险动作需要填写原因。`直接修订` 明确意味着审核员在页面内通过 GUI 表单编辑标注答案，字段优先按模板 schema 渲染为单选、多选、数字、开关或文本输入；复杂对象字段才退化为单字段 JSON 输入。审核员填写修订说明后直接入库，后端记录修订前后答案和审计痕迹，不是静默覆盖。

### 6.4 右侧概览栏

右栏是 Reviewer 的个人与工作概览，不是任务详情补充区。

建议内容：

- 今日已审数量
- 今日通过率
- 待审数量
- 涉及任务数
- 当前选中提交的审计时间线

右栏信息应该紧凑，顶部固定 4 个数字块，下方展示当前选中提交的审计时间线；不要堆大卡片。审计时间线按 `round` 分组，每个分组用浅色虚线框呈现并整块可点击。分组内可包含标注员提交、AI Agent 预审、Reviewer 复核/打回/通过等同轮次日志；每条日志首行展示账号名与提交/操作时间，下方展示操作摘要；点击分组即切换中间详情的审核轮次视角。

## 7. 关键交互

### 7.1 队列切换

- 切换 `AI建议通过`、`AI建议打回`、`待人工审核` 时，只刷新左侧列表和中间详情，不重置整页布局。
- 队列切换后保留当前滚动位置附近的稳定感，不做页面抖动。

### 7.2 批量处理

- 勾选多条后显示批量操作条。
- 批量通过可直接执行。
- 批量打回必须输入统一原因或选择原因模板。
- 批量转交必须选择目标 Reviewer 或组。

### 7.3 单条审核

- 打回和修订必须留痕。
- 通过入库后状态应立即更新到队列和右侧概览。
- 下一条按钮应自动切换到同队列下一条待处理项。

### 7.4 历史与审计

- 审核历史和审计时间线应可展开查看。
- 审计时间线默认在右侧栏按轮次分组展示，点击分组切换当前审核视角；该交互替代标题栏阶段切换。
- 默认只露出最近几条或按右侧栏内部滚动，避免中间栏被时间线撑爆。
- 导出审计日志入口应与当前页面筛选条件联动。

## 9. 视觉与状态

- 主色延续 MarkUp 工作台浅蓝白灰体系。
- AI 预审结果区使用浅蓝提示。
- 打回、风险和待复审使用橙红色语义。
- 通过使用绿色语义。
- 批量选择态、当前选中态和焦点态必须清晰，但不要出现重阴影和大圆角卡片堆叠。

加载和空态：

- 页面级加载使用 `WorkspaceLoading` 或 `Spin`。
- 队列空态使用 `Empty` + 简短说明。
- 权限不足使用 `Alert`。
- 审核提交中按钮必须 `loading`。

## 10. 权限与降级

权限规则建议：

- Reviewer 默认仅查看分配范围或 `assigned_only` 队列。
- Team Admin / Owner 可查看企业范围。
- 未分配数据不可越权查看。
- 批量操作需要后端权限校验，不依赖前端隐藏。

降级规则：

- 如果 AI 预审结果缺失，详情区应显示“等待 AI 结果”或类似状态，不伪造分数。
- 如果历史轮次不完整，明确提示“仅展示当前可用轮次”。
- 如果审计日志未拉到，应保留详情内容并提示稍后重试。

## 11. 与验收图的对应关系

`docs/assets/验收.png` 展示的是典型的三栏审核工作台骨架，当前设计稿在此基础上做了以下收敛：

- 左栏从“AI 建议通过 / 打回 / 转人工”扩展为完整的队列筛选与批量处置区。
- 中栏从“单条审核面板”扩展为原始数据、标注结果、AI 结果、历史与最终动作的组合区。
- 右栏从单纯统计卡扩展为 Reviewer 账号概览与工作摘要。
- 顶部固定在一个浏览器视口内，避免长页面滚动导致的上下文丢失。

## 12. 交付约束

- 页面设计必须同步更新到统一设计规范和页面索引。
- 后续实现时不得引入新的 UI 库。
- 页面不能退化成两栏或多卡片堆叠式布局。
- 批量、详情、历史、审计和右侧概览都必须保留独立职责。
