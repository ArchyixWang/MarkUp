# 资源配置页面设计稿

## 1. 页面定位

资源配置承载企业级生产资源管理：会员与额度、企业积分钱包、AI 资源、AI Provider 和生产资源开关。它是企业管理区的配置中枢，服务 Team Admin、Owner、Agent 和平台管理员。

页面主线固定为三条：

- `会员与额度`：企业会员管理入口，负责套餐、有效期、三项资源用量和购买/续费/预约降级。
- `积分管理`：企业积分钱包治理入口，负责余额观察、模拟充值、预警设置和钱包流水导出。
- `AI 资源`：AI 调用观察入口，展示 AI 钱包摘要、Token、成本估算、Provider 状态与统一调用历史，不承接预算语义。

## 2. 信息架构

页面结构：

```text
资源配置
├─ 固定标题栏
├─ Tabs
│  ├─ 会员与额度
│  ├─ 积分管理
│  ├─ AI 资源
│  ├─ AI Provider

├─ 配置 Drawer
└─ 日志入口
```

标题栏下不再保留单独的首行统计栏，页面直接进入 Tabs。

页面右上动作固定为：

- `刷新`
- `查看操作日志`

`配置积分池` 这类头部独立按钮不再保留，会员购买、充值和预警操作收敛到对应 Tab 内。

## 3. 会员与额度 Tab

### 3.1 首屏布局

会员与额度是资源配置页默认 Tab，使用紧凑企业后台布局：

1. 当前套餐摘要：当前有效套餐、有效期、账面套餐、预约降级状态、企业钱包可用余额。
2. 超额提示：只有 `over_limit_items` 非空时展示 warning。
3. 三项用量：成员、活跃生产任务、数据集存储，使用进度条或紧凑数值卡；成员用量不包含企业内置系统 Agent。
4. 套餐卡片：Free、Basic、Pro、Enterprise、More。

### 3.2 套餐交互

- `Free`：当前非 Free 且仍在有效期内时展示 `预约降级`，提交后写 `membership_next_plan=free`。
- `Basic / Pro / Enterprise`：展示年费、成员上限、活跃任务上限和数据集存储；点击 `购买 / 续费` 打开 Drawer。
- `More`：只展示“联系平台定制”，点击打开说明 Modal，不调用购买接口、不扣费、不落套餐字段。
- 余额不足时购买按钮保持可见，但 Tooltip 和 Drawer 内 warning 提示先去积分管理充值。
- 未设置企业钱包支付密码时不打开购买 Drawer，提示先到积分管理设置支付密码。

### 3.3 购买 Drawer

购买/续费 Drawer 必须展示：

- 目标套餐
- 年费积分
- 企业钱包可用余额
- 企业钱包支付密码输入框

提交 payload 固定为：

```json
{
  "target_plan": "basic",
  "payment_password": "******"
}
```

V1 不提供优惠、折算、自动续费或发票流程；这些不应在页面上提前展示。

## 4. 积分管理 Tab

### 4.1 首屏布局

积分管理 Tab 必须是极限压缩结构：

1. 单行钱包摘要
2. 单行操作条
3. 钱包流水表

不再保留“大积分预算治理卡”。

### 4.2 钱包真实口径

四个主状态字段不是重命名，而是按真实业务含义展示：

- `积分余额 = balance_points`
- `预扣积分 = reserved_points`
- `花销统计 = spent_points`
- `可用余额 = available_points`

约束：

- `预扣积分` 只统计已发布任务的奖励占用；任务一经发布即预扣，不需要先被领取。
- `可提现余额` 与 `可用余额` 完全一致，预扣中的积分不能用于提现。
- 钱包摘要只显示标签和大数字，不再显示每列下方的小字说明。
- 主值使用紧凑数值显示，例如 `1.2 万`、`3.4 亿`、`1.1 万亿`。
- Tooltip、Drawer 摘要和详情位显示完整千分位数字。

### 4.3 操作条

操作条固定为：

- `积分充值`
- `积分提现`
- `支付密码`
- `预警设置`
- `积分审计`

实现约束补充：

- `支付密码` 入口前端按钱包查看/管理能力可见，避免管理员被误隐藏；提交权限仍以后端 `team_admin_only` 为准。
- 钱包流水表上方保留紧凑筛选条：`类型 / 方式 / 状态 / 关键词 / 时间范围 / 导出流水`。
- `导出流水` 默认导出当前筛选结果，支持 `CSV / Excel / JSON`。
- 充值金额录入与确认页都必须明确展示 `1 积分 = 1 元`。

状态区只保留一行紧凑语义：

- `余额充足`
- `接近预警`
- `可用余额耗尽`

同时展示当前预警是否开启及阈值。

### 4.4 充值 Drawer

充值 Drawer 为分支式模拟支付流程，不使用通用确认弹窗糊过去。

流程：

1. 选择支付方式
2. 填写充值积分
3. 进入方式专属确认界面

支付方式固定为：

- `微信支付`：`WechatOutlined`
- `支付宝`：`AlipayCircleOutlined`
- `对公转账`：`BankOutlined`

方式专属界面要求：

- 微信：二维码区域、订单号、金额、有效期、`我已完成支付`
- 支付宝：支付摘要、跳转支付按钮态、`支付完成`
- 对公转账：收款主体、开户行、账户名、银行账号、转账附言、到账说明、复制按钮、`我已完成转账`

业务约束：

- 本轮仍是模拟支付，不接真实支付网关。
- 最终持久化动作仍是企业积分钱包充值接口。
- 支持万亿级积分录入，但必须限制在 JS 安全整数范围内的业务上限内。

### 4.5 预警 Drawer

字段固定为：

- `是否开启预警`
- `使用率阈值`

说明文案固定为“按可用余额预警”，阈值填写实际积分值。

### 4.6 钱包流水表

积分管理页底部表格已从“任务占用来源表”收敛为“企业积分钱包流水表”，用于展示企业钱包的真实资金流向，而不是任务配置分析。

表格定位：

- 展示企业钱包每一笔入账、出账和审核结算扣减
- 让 Team Admin / Owner 直接核对充值、提现、奖励支出后的余额变化
- 作为“查看积分审计”之外的业务流水视图

建议列：

- 流水时间
- 类型
- 收支
- 余额
- 来源/方式
- 状态
- 备注

当前流水类型至少包含：

- `recharge`
- `withdraw`
- `reward_spend`

说明：

- 流水表记录的是“钱包账户变化”
- 审计日志记录的是“谁在什么时候执行了什么操作”
- 两者都保留，但不能混为同一张表

## 5. AI 资源 Tab

### 5.1 首屏布局

AI 资源改成条形观察页：

1. 单行紧凑指标
2. AI 钱包紧凑摘要
3. 单行操作条
4. 一条简短说明
5. 统一调用历史表

不再保留大观察卡、独立调用日志 Drawer 和长说明段。

### 5.2 指标与动作

指标固定为：

- `累计 Token`
- `本月成本`
- `最近调用数`
- `可用 Provider`

AI 钱包摘要只保留：

- `AI 钱包余额`
- `状态 Tag`
- `更新时间`

操作固定为：

- `AI 积分充值`
- `估算成本`
- `查看 AI 审计`

估算成本放入独立 Drawer，不在首屏常驻展开。

补充约束：

- `AI 积分充值` 的真实语义是“企业积分钱包向 AI 调用钱包划转”。
- 划转时必须输入企业当前 6 位支付密码。
- AI 积分充值 Drawer 不展示微信、支付宝、对公转账或支付金额，只展示企业钱包可支配余额、转入积分、扣减账户、转入账户和支付密码。
- 历史表统一展示 `transfer_in / ai_call / adjustment`，由后端归一化返回。

### 5.3 文案边界

必须明确：

- 真实消耗发生在 Provider 小号侧。
- 平台只记录调用日志、Token 和成本估算。
- 平台共享路由会实扣 AI 钱包，自配 Provider 只做记录与估算。
- 当前页不出现预算、充值、预警、审批、剩余额度等 AI 预算语义。

## 6. Agent 设置

`Agent 设置` 仍保留，但不再占用 `积分管理` 首屏。

当前建议放置在 `AI Provider` 相关区域，通过独立按钮打开 Drawer。

约束：

- 仅 Team Admin 可编辑
- 支持官方预设头像、自定义上传和恢复默认；自定义头像上传控件仅接受 JPG、PNG、GIF，并在前端拦截非图片文件后再调用上传接口
- 不提供删除、禁用、改角色或改用户名入口

## 7. 数值显示规则

资源配置页中的大数值统一遵守：

- 主展示位：紧凑格式
- Tooltip / Drawer / 详情：完整千分位格式
- 充值输入：只接收整数数字串，前后端共同限制安全上限

适用范围至少包括：

- 钱包摘要四项
- 充值 Drawer 摘要
- 钱包流水表
- 充值成功反馈
- AI Token 指标

## 8. API 对应

会员与额度：

- `GET /api/v1/teams/{team_id}/membership`
- `POST /api/v1/teams/{team_id}/membership/subscribe`
- `POST /api/v1/teams/{team_id}/membership/cancel-scheduled-change`

积分管理：

- `GET /api/v1/teams/{team_id}/points-budget`
- `GET /api/v1/teams/{team_id}/points-budget/ledger`
- `POST /api/v1/teams/{team_id}/points-budget/recharge`
- `POST /api/v1/teams/{team_id}/points-budget/withdraw`
- `POST /api/v1/teams/{team_id}/points-budget/alerts`

AI 资源：

- `GET /api/v1/ai-resources/teams/{team_id}/reports/cost`
- `GET /api/v1/ai-resources/teams/{team_id}/wallet`
- `GET /api/v1/ai-resources/teams/{team_id}/history`
- `POST /api/v1/ai-resources/teams/{team_id}/wallet/transfer-in`
- `POST /api/v1/ai-resources/estimate`
- `GET /api/v1/ai-resources/configs`

## 9. AI Provider 配置中心补充

当前 `AI Provider` Tab 已收敛为“单路由单模型”的专业配置中心，不再按“一个 Provider 挂多个模型”的心智设计。

一条 Provider 记录固定代表：

- 一个 `route_name`
- 一个 `provider_kind`
- 一个 `model_id`
- 一套 `Base URL / API Key / 运行参数`
- 一组显式 `capabilities`
- 一组显式 `pricing`

页面结构固定为：

- 左侧配置列表：展示配置名称、Provider 类型、模型、状态、费率摘要、最近测试状态
- 右侧详情面板：展示基础信息、鉴权与接入、运行参数、价格配置、能力声明和最近测试结果
- 宽 Drawer：承接新增与编辑

企业侧展示补充：

- 平台共享 Provider 在企业侧只展示平台配置昵称 `route_name`、价格摘要和模态能力。
- 平台共享 Provider 不展示 API Base、API Key 状态、运行参数、最近测试详情或 request_id。
- 右侧详情矩阵只服务企业自配 Provider；选中平台共享 Provider 时只保留名字、价格和模态能力，不展开任何配置细节。

页面动作固定为：

- `新增配置`
- `编辑`
- `复制`
- `启用/停用`
- `删除`
- `测试连接`

费率字段固定为：

- `每百万输入价格`
- `每百万输出价格`
- `每百万 Cache 命中价格`

API Key 交互固定为：

- 新增时可直接录入
- 编辑时默认不回显历史明文
- 可通过密码框显式切换可视/隐藏
- 编辑未填新 Key 时保持原密钥不变
- 编辑填入新 Key 视为轮换

`AI Provider` Drawer 继续按 Ant Design 的企业后台密度实现，但不再使用“所有 Provider 共用同一套接入表单”的形式。当前页面应按 `provider_kind` 动态切换厂商专属接入字段：

- `OpenAI`：补充 `Organization ID / Project ID`
- `OpenAI Compatible`：补充 `自定义请求头(JSON)`
- `DeepSeek`：使用官方 Base URL 与模型 ID，不额外暴露兼容网关字段
- `OpenRouter`：补充 `站点地址 / 应用名称`
- `Anthropic`：补充 `Anthropic Version`
- `Gemini`：补充 `API Version`
- `Azure OpenAI`：补充 `Azure 资源名 / API Version`，并在前端联动生成 Azure Endpoint
- `Ollama / LM Studio`：补充 `Keep Alive / 上下文窗口`
- `通义千问`：补充 `Workspace ID`
- `方舟`：补充 `Region`

新增/编辑 Drawer 内必须直接提供检测动作，而不是要求用户先离开 Drawer 再去详情区操作。当前主路径收敛为：

- `仅创建 / 仅保存`
- `创建并检测 / 保存并检测`

其中“并检测”走真实后端测试接口，创建模式下先落库再执行连通性检测，检测结果同步回写到列表与详情面板。

企业侧平台共享路由展示补充：

- 平台共享 Provider 卡片与只读详情主标题统一显示平台配置的 `route_name`。
- 企业侧不直接展示平台共享路由的 Provider 类型、兼容协议类型或底层实际调用模型名，避免暴露实现细节。
- 企业页与平台页的 Provider 动态字段、默认值、费率摘要、测试状态文案必须复用同一份前端共享元数据源，避免出现“平台页可测、企业页字段不生效”或不同页面 Provider 契约漂移。
- 企业侧 `AI Provider` Drawer 与平台侧新增/编辑 Drawer 都必须内置 `测试连接`，并按是否存在未保存改动自动切换到草稿测试或已保存配置测试。

## 10. 设计约束

- 全部复杂交互优先使用 Ant Design 原生组件。
- 所有异步按钮必须提供 loading。
- 错误提示尽量贴近表单或卡片，不只依赖全局 message。
- 高风险动作必须二次确认。
- 移动端退化为单列卡片堆叠，不允许核心操作落到横向滚动区域之外。
## 2026-05-31 平台共享 Provider 与企业 AI 钱包补充

### 页面基线

- `AI 资源` 已升级为“观察 + 平台共享路由计费入口”。
- 企业奖励积分钱包与 `AI 调用积分钱包` 是两套独立账。
- AI 钱包文案统一使用 `AI 积分`；AI 积分充值是企业积分钱包内部划转，不再展示 `1 积分 = 1 元`、微信、支付宝或对公转账。

### AI 资源

- Tab 顶部增加紧凑的 `AI 调用积分钱包` 摘要，展示余额、状态与更新时间。
- `AI 积分充值` 改为从企业积分钱包原子划转到 AI 钱包，最终确认时必须输入支付密码；请求体只包含 `amount` 和 `payment_password`。
- 页面底部主表统一改为 `调用历史`，合并 AI 钱包转入流水与 AI 调用日志，不再保留单独的 AI 流水主路径。
- `测试连接`、`估算成本` 和 `AI 审计` 不扣 AI 钱包。

### AI Provider 列表与详情

- 平台可维护多条 `scope=platform` 的共享路由。
- 全局最多仅允许一条平台路由标记为 `is_platform_default=true`。
- 企业视角下 Provider 列表同时展示企业自有路由与平台共享路由。
- 平台共享路由在企业侧必须显示只读标识，不允许企业修改密钥、接入地址和费率。
- 平台共享路由在企业侧不展示接入细节，只展示平台昵称、价格和模态能力。
- `is_platform_default=true` 的平台路由显示 `平台默认` 标签。
- 平台共享路由显示 `平台共享` 标签。

### 调用选择约束

- 企业在任务发布页和其他 AI 调用入口可以直接选择平台共享路由。
- 平台默认路由只提供默认候选语义，不自动预选。
- 仅当所选路由 `scope=platform` 时，才会消耗企业 `AI 调用积分钱包`。
- 企业自有 Provider 继续只做日志记录与成本估算，不从平台侧 AI 钱包扣费。
- 2026-06-01 补充：企业侧 `AI Provider` 页面继续朝高密度企业后台方向收口；平台共享 Provider 卡片与详情进一步降噪到仅展示 `route_name` 作为唯一主名，并保留价格与模态能力，企业自配 Provider 的右侧详情继续使用更小卡片内边距与双列 `Descriptions`，在同屏内呈现更多核心配置。
- 2026-06-01 补充：企业侧 `AI Provider` 内的平台共享区块改为“平台共享路由”，选择后右侧详情只展示平台填写的业务昵称、价格和模态能力，不再重复出现“平台共享 Provider”式标题，也不暴露底层接入或测试细节。
