# 企业管理页面组设计稿

## 1. 页面组定位

企业管理是企业/企业维度的治理中心，覆盖企业资料、资源与预算、消息分发和审计追踪。它不直接承载任务生产流程，但必须为任务发布、模板搭建、标注、AI 预审、人工审核和导出提供企业级前置条件。

页面组包含四个独立页面：

| 页面 | 职责 | 不承担 |
| --- | --- | --- |
| 企业信息 | 认证、展示和修改企业基础信息。 | 不管理成员列表、预算明细、消息和日志。 |
| 资源配置 | 企业积分钱包、AI 资源、AI Provider 和生产资源开关。 | 不直接执行 AI 调用业务流程，不审批个人资质，不伪造企业积分池余额。 |
| 公告通知 | 系统公告、企业通知、审核提醒和导出完成提醒。 | 不做聊天工具，不替代审核/导出业务页面。 |
| 操作日志 | 关键状态流转、权限变更、审核、导出和企业管理审计日志。 | 不提供直接修改业务状态的操作。 |

设计目标：

- 让 Team Admin / Owner 能快速完成企业级配置和治理。
- 让 Reviewer、Agent、Labeler 只看到与自己权限相关的信息，避免误操作入口。
- 让所有高风险变更可追溯，并能跳转到操作日志。
- 让缺失接口以只读、禁用或待接入方式降级，不伪造认证、成本、分发人数和审计事实。

企业管理页组的设计优先级按用户使用频率和风险排序：

1. 先保证企业资料、认证状态、预算状态和通知未读数可被快速确认。
2. 再保证 Team Admin / Owner 能完成资料保存、积分预算治理、通知发送和日志追溯。
3. 最后补齐成本报表、自动提醒、深链接、异步导出和实时推送等增强能力。

前端实现时不要把四页做成同一套“标题 + 卡片 + 表格”的通用模板。企业信息以资料与认证为核心，资源配置以配置分区为核心，公告通知以分发和处理为核心，操作日志以查询和追溯为核心。

本轮设计按 `ui-ux-pro-max -> hallmark -> Ant Design` 链路收敛：先按企业治理后台定义信息架构，再检查四页是否出现同质化模板，最后落到 Ant Design 的表格、表单、Tabs、Drawer、Modal 和 Descriptions 组件。四页共享导航和标题栏，但主信息形态不同：

- 企业信息以资料表单和认证状态为主。
- 资源配置以分区 Tabs、配置表格和预算治理/资源观察双主线为主。
- 公告通知以消息表格、接收人构建器和分发预览为主。
- 操作日志以审计筛选表格、详情 Drawer 和 diff 为主。

## 1.3 本轮讨论确认的设计边界

根据当前交付文档第 4 章和企业管理讨论，四页要服务企业级治理，而不是复制生产页或账号设置页：

| 页面 | 必须强化 | 必须避免 |
| --- | --- | --- |
| 企业信息 | 企业认证、企业资料展示与修改、认证材料、资料变更审计。 | 做成营销式企业主页，或把人员、预算、通知配置塞入同页。 |
| 资源配置 | 企业积分钱包、AI 资源、AI Provider、生产资源开关。 | 让业务页面直接保存 provider secret、伪造积分池余额，或只做前端本地开关。 |
| 公告通知 | 系统公告、企业通知、审核提醒、导出完成提醒，按企业人员结构分发。 | 做成聊天窗口，伪造自动提醒、邮件发送状态或精确接收人数。 |
| 操作日志 | 关键状态流转、权限变更、审核、导出和企业管理审计日志。 | 提供绕过业务页面的修改按钮，或把日志做成不可筛选的时间线。 |

页面实现需要保持“固定标题栏 + 主体自然滚动”的工作台模式。标题栏固定在面包屑下方，不随内容滚动；长表格、长表单、详情和历史记录通过分页、Drawer、Modal 或子页面承载，不强行压缩到一屏。

## 1.1 目标用户与主要任务

| 用户 | 高频任务 | 设计侧重点 |
| --- | --- | --- |
| Team Admin | 维护企业资料、认证、成员治理、资源开关、审计追踪。 | 显示完整治理入口，强调权限、风险确认和审计回溯。 |
| Owner | 保障任务发布所需的预算、Provider、资质和通知协作。 | 将资源配置和生产链路影响关系讲清楚，避免发布时才发现缺口。 |
| Reviewer | 接收审核提醒、查看与自己相关的通知和部分操作记录。 | 限制企业级改动入口，只保留相关提醒和跳转。 |
| Agent | 查看 AI 资源观察结果、模型限制和相关日志。 | 不暴露 Provider secret，不把历史预算申请流作为当前主路径。 |
| Labeler | 接收企业公告、打回提醒和导出无关的个人通知。 | 只呈现个人相关消息，不暴露企业审计和资源配置。 |

## 1.2 企业治理对象

四页围绕同一组企业级对象构建，前端实现时应保持数据字段和筛选口径一致：

| 对象 | 来源 | 主要使用页面 |
| --- | --- | --- |
| 企业资料与认证 | `teams` | 企业信息、操作日志。 |
| 成员与角色 | `team_members` | 公告通知分发、资源审批人、操作日志筛选。 |
| 钱包与 AI 资源 | `teams.budget`、`ai_resources` | 资源配置、公告通知预算提醒、操作日志。 |
| 资质类型 | `cert_types` / AI 资源聚合 | 任务发布资质引用、平台侧资质治理。 |
| 通知记录 | `notifications` | 公告通知、操作日志。 |
| 审计记录 | `audit_logs` | 操作日志、各页“查看日志”入口。 |

## 2. 导航与信息架构

侧栏分组建议：

```text
企业管理
├─ 企业信息
├─ 人员管理
├─ 资源配置
├─ 公告通知
└─ 操作日志
```

四个页面都使用工作台统一固定标题栏：

```text
工作台 / 企业信息
工作台 / 资源配置
工作台 / 公告通知
工作台 / 操作日志
```

标题栏固定在全局导航和面包屑下方，不随表格或表单内容滚动。页面主体允许自然纵向滚动；长列表、表格和详情抽屉使用 Ant Design 的分页、筛选和 Drawer 承载，不把所有内容强行压缩到一屏。

## 3. 公共布局模式

企业管理页面统一采用以下结构：

```text
固定页面标题
├─ 页面名称、简短说明
├─ 主操作 / 刷新 / 查看日志
└─ 当前页状态栏或关键摘要

页面主体
├─ 筛选条 / Tabs / 表单分组
├─ 表格 / Descriptions / Form
├─ Drawer / Modal
└─ 空态、错误态、权限提示
```

设计约束：

- 以表格、表单、描述列表为主要信息形态，少用大面积卡片墙。
- 页面主操作只保留一个高优先级按钮，其余操作放到次级按钮、更多菜单或行内操作。
- 所有危险操作使用 `Popconfirm` 或 `Modal.confirm` 二次确认。
- 所有涉及企业作用域的 API 必须传递并校验 `X-Team-ID`。
- 所有企业级写操作必须写入审计日志，前端提供跳转入口。
- 固定标题栏只承载页面身份、关键操作和少量摘要，表格筛选和长内容在主体内滚动，避免标题随着内容滚动丢失上下文。
- 同一页内的 Drawer / Modal 只承载当前操作，不复用为跨页详情容器；跨页查看使用工作台内部跳转或后续 URL 查询参数。

## 4. 企业信息

详细设计见 `organization-profile.md`。

核心内容：

- 企业状态摘要：企业名称、企业状态、认证状态、行业、成员数、AI 资源观察摘要。
- 认证区：展示认证主体、统一社会信用代码、联系人、材料、审核意见和最近提交时间。
- 基础资料表单：企业名称、行业、联系电话、官网、地址、简介、Logo。
- 成员与角色统计：Team Admin、Owner、Reviewer、Agent、Labeler 分布。
- 操作日志入口：默认筛选当前企业或企业资料变更。

关键规则：

- Team Admin / Owner 可编辑基础资料；普通成员只读。
- 企业认证是平台运营能力，Team Admin 不能自审认证。
- 已认证后修改主体名称、证件号、认证联系人等敏感字段必须提示可能重新认证。
- 认证提交接口已接入时展示真实 `verification_status`；平台审核回写未接入时不能伪造已认证状态。

## 5. 资源配置

详细设计见 `organization-resource-config.md`。

资源配置使用 Tabs 分区：

| Tab | 内容 |
| --- | --- |
| 企业积分钱包 | 积分余额、预扣积分、花销统计、可用余额、充值、提现、支付密码、预警规则与钱包流水。 |
| AI 资源 | AI 钱包摘要、AI 积分充值、估算成本、统一调用历史。 |
| AI Provider | 单路由单模型配置、价格、模态能力、测试连接与编辑配置。 |
| 生产开关 | 任务发布、AI 预审、LLM 辅助、数据导出、上传、公开任务广场。 |

关键规则：

- AI 调用必须通过 AI Resources / AI Gateway，业务页面不直接保存或调用 provider secret。
- API key 不回显明文，编辑时留空表示不变。
- 资质类型是平台级能力，不再作为资源配置页内的独立 Tab；Team Admin / Owner 只在任务发布等业务页引用结果，不能审批 Labeler 资质。
- 关闭 AI 预审、导出、上传、任务发布等高风险开关必须二次确认，并写操作日志。
- 成本报表、AI 钱包统一历史或 Provider 写接口缺失时只读降级，不用假数据补齐。

## 6. 公告通知

详细设计见 `organization-announcements.md`。

公告通知以表格为主视图，按类型分为：

- 系统公告：平台版本、维护、安全和规则通知。
- 企业通知：Team Admin / Owner 创建，面向企业成员分发。
- 审核提醒：待审核、AI 预审异常、打回修改、复审/终审提醒。
- 导出提醒：导出创建、完成、失败和下载即将过期。

通知分发对象：

- 全企业。
- 按角色：Owner、Reviewer、Agent、Labeler。
- 指定成员：复用人员管理成员列表。
- 指定任务相关成员：任务 Owner、Reviewer、Agent、Labeler 贡献者。
- 指定审核队列成员：assigned reviewer。

关键规则：

- 新建企业通知必须先预览接收人，0 人时禁止发送。
- 紧急通知发送前必须二次确认。
- SMTP 未启用时仅保留站内通知，不显示邮件已发送成功。
- WebSocket 不可用时通过 REST 刷新或轮询降级。
- 当前任务相关人员聚合未完成时，按任务分发必须明确提示临时降级策略，不能伪造精确分发人数。

## 7. 操作日志

详细设计见 `organization-audit-logs.md`。

操作日志主视图是可筛选表格，覆盖：

- 状态流转：任务发布、暂停、恢复、结束；模板发布；导出创建、完成、失败、下载。
- 权限变更：添加成员、邀请成员、更新角色、禁用/移除成员、权限集合变化。
- 审核：人工通过、打回、要求修改、批量审核、查看 diff。
- AI 与预算：Provider 配置、AI 钱包划转、AI 资源观察，以及企业积分钱包治理。
- 数据与导出：数据集导入、模板 schema 变更、导出下载、取消导出。
- 通知与企业：创建/撤回企业通知、修改企业信息、提交认证。

关键规则：

- 日志详情 Drawer 必须展示 request_id、操作人、实体、风险等级、变更摘要、diff 和关联对象。
- API key、密码、验证码、token 和签名下载链接必须脱敏。
- 风险等级如果由前端临时推断，必须标注为展示层逻辑；最终口径应由后端写入审计字段。
- 导出审计日志本身也必须写入审计日志。

## 8. 跨页面联动

| 来源页面 | 动作 | 目标 |
| --- | --- | --- |
| 企业信息 | 查看操作日志 | 操作日志，筛选 `entity_type=team`。 |
| 企业信息 | 点击角色统计 | 人员管理，带角色筛选。 |
| 企业信息 | 点击 AI 资源摘要 | 资源配置 / AI 资源。 |
| 资源配置 | 查看 AI 审计 | 操作日志，筛选 `entity_type=ai_resource`。 |
| 资源配置 | 查看操作日志 | 操作日志，筛选 `entity_type=ai_resource` 或 `resource_config`。 |
| 人员管理 | 查看成员操作日志 | 操作日志，筛选 `entity_type=team_member`；成员详情入口额外携带 `entity_id=user_id`。 |
| 公告通知 | 点击审核提醒 | 审核工作台或任务审核详情。 |
| 公告通知 | 点击导出提醒 | 导出中心或导出任务详情。 |
| 操作日志 | 点击实体 | 跳转到任务、模板、数据集、成员、导出等对应详情。 |

深链接暂未完整实现时，前端可先通过内部页面状态跳转；但设计文档和后续路由应保留 `entity_type`、`entity_id`、`operator_id`、`tab` 等参数。

建议查询参数：

| 参数 | 用途 |
| --- | --- |
| `tab` | 资源配置、公告通知等页面的默认页签。 |
| `entity_type` | 操作日志默认实体类型筛选。 |
| `entity_id` | 操作日志默认实体 ID 筛选。 |
| `operator_id` | 操作日志默认操作人筛选。 |
| `role` | 人员管理默认角色筛选。 |
| `notification_type` | 公告通知默认消息类型筛选。 |

当前前端已通过工作台内部状态支持企业信息、资源配置和人员管理跳转操作日志时携带 `entity_type` / `entity_id` 初始筛选；后续如果路由拆成子路由，应把这些字段迁移到 URL 查询参数，避免重写交互。

## 10. 状态与降级

加载：

- 页面首屏使用 `Skeleton` 或表格 loading。
- Tabs 独立 loading，不因某个配置接口失败阻塞全页。
- 保存、审批、发送、开关变更使用局部 loading。

空态：

- 无企业时引导创建企业或返回 onboarding。
- 无 Provider、无积分预算数据、无通知、无日志时保留表头和筛选能力。
- 无权限时展示权限提示，不渲染可误操作按钮。

错误：

- 失败信息展示后端字段级原因和 request_id。
- 表单提交失败保留用户输入。
- 创建通知、审批预算、关闭开关失败后恢复原状态。

接口缺口降级：

| 缺口 | 降级方式 |
| --- | --- |
| 企业认证审核回写未接入 | 提交后展示 `审核中`，不提供 Team Admin 自审入口。 |
| Logo / 材料上传未接入 | Logo 上传 disabled；认证材料先提交 URL 列表。 |
| Provider 写接口未接入 | Provider 表格只读，隐藏编辑。 |
| 企业级通知策略未接入 | 个人通知偏好先复用 `/profile/me.notification_settings`；企业强制通知、邮件策略和实时订阅仍展示说明或只读降级。 |
| WebSocket 未接入 | 手动刷新或轮询。 |
| 审计异步导出未接入 | 当前页已接入后端同步 CSV 导出；超大范围导出、进度和下载历史后续接异步任务。 |

## 11. 当前实现边界

企业管理四页的设计必须同时区分“当前可用能力”和“后续目标能力”：

| 页面 | 当前可用能力 | 后续目标 |
| --- | --- | --- |
| 企业信息 | 展示默认企业、保存基础资料、提交企业认证并显示认证进度。 | 认证材料上传、认证审核状态回写、撤回认证。 |
| 资源配置 | 积分预算治理、AI 资源、Provider 列表、调用日志、生产开关。 | 企业积分池冻结/扣减/冲正接口、真实 AI Gateway、Provider 密钥加密调用、更细成本报表、批量 AI 调用。 |
| 公告通知 | 企业通知列表、创建、分发预览、已读/处理状态、撤回、软删除、个人通知偏好保存。 | 系统公告、审核提醒、导出提醒自动写入、企业级通知策略、WebSocket 实时推送。 |
| 操作日志 | 企业作用域日志、关键词/实体/风险/时间筛选、详情、当前筛选服务端 CSV 导出。 | 异步审计导出、风险等级持久化、关联对象名称冗余、完整 request_id 历史回填。 |

页面文案要避免把后续目标描述成已上线能力。未接入能力使用 disabled、说明 Drawer 或只读占位，不能通过前端假数据补齐。

## 11.1 前端实现优先级与页面差异化

企业管理剩余四页建议按以下顺序实现和验收：

| 顺序 | 页面 | 原因 | 最小可交付 |
| --- | --- | --- | --- |
| 1 | 企业信息 | 企业资料、认证状态和企业作用域是后续资源、通知、日志的基础。 | 企业资料展示/保存、认证提交、成员/预算摘要和日志跳转。 |
| 2 | 资源配置 | 直接影响任务奖励、任务发布、AI 预审、LLM 辅助、导出和上传。 | 企业积分钱包、AI 资源、AI Provider、生产开关四个主配置区可读可操作。 |
| 3 | 公告通知 | 承载成员结构上的消息分发和审核/导出提醒。 | 企业通知列表、分发预览、发送、已读/处理、通知设置。 |
| 4 | 操作日志 | 为前三页和生产链路提供可追溯性。 | 筛选表格、详情 Drawer、diff、当前筛选导出和跨页来源筛选。 |

四页共享工作台标题栏、面包屑和基础间距，但主体结构必须不同：

| 页面 | 主信息形态 | 主要操作承载 | 详情承载 |
| --- | --- | --- | --- |
| 企业信息 | 左侧资料表单 + 右侧治理栏。 | 表单保存、认证提交。 | 认证材料 Drawer、最近变更跳日志。 |
| 资源配置 | Tabs + 配置表格。 | 积分充值、提现、支付密码、Provider 编辑、开关切换。 | Provider / AI 钱包划转 / 积分钱包 / 预警 Drawer。 |
| 公告通知 | 消息工具条 + Tabs + 消息表格。 | 新建企业通知、批量已读/处理。 | 消息详情 Drawer、接收人预览 Drawer。 |
| 操作日志 | 筛选工具条 + 审计表格。 | 导出日志、复制 request_id。 | 日志详情 Drawer、diff 面板。 |

实现时先落地真实接口已有能力；后续能力通过 disabled、`待接入` 或只读说明表达。不要为了视觉完整而填充演示成本、接收人数、认证结果、IP、设备或审计事实。

## 12. Ant Design 组件基线

- `Table`：成员、资源、通知、日志主列表。
- `Descriptions`：企业摘要、日志详情。
- `Form`：企业资料、认证、积分充值/预警、Provider 编辑。
- `Tabs`：资源配置和通知类型切换。
- `Drawer`：详情、编辑、钱包划转、通知设置。
- `Modal` / `Popconfirm`：危险确认、紧急通知、删除确认。
- `Tag` / `Badge` / `Progress`：状态、未读、钱包状态和风险等级。
- `Upload`：Logo、认证材料，未接入时 disabled。

## 13. 验收标准

- 企业管理侧栏下四个页面职责清晰，互不混塞。
- 四页都使用固定标题栏和 Ant Design 中后台组件，不做营销式大卡片布局。
- 企业信息能展示和编辑基础资料，并支持提交企业认证；审核回写和材料上传缺口需明确降级。
- 资源配置能覆盖企业积分钱包、AI 资源、AI Provider 和生产开关。
- 公告通知能按人员结构进行企业通知分发，并承载系统、审核、导出提醒。
- 操作日志能查询关键状态流转、权限变更、审核、导出和企业管理审计。
- 所有高风险操作有确认、权限限制和审计入口。

## 14. Hallmark 检查

- 不把企业管理做成设置卡片堆叠。
- 不让四个页面共用一套只有标题不同的模板；每页的主信息形态必须匹配职责。
- 不伪造认证状态、成本、通知分发人数、IP、设备和审计风险。
- 不在通知页做聊天流，不在日志页做时间线卡片流作为主视图。
- 不绕过服务端权限、AI Gateway、预算控制和审计写入。
